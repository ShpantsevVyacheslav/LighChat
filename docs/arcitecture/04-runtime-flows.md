[`RegisterAvatarCropOverlay`](../src/components/auth/register-avatar-crop-overlay.tsx) дополнительно экспортирует круг ([`getCircularCroppedImageBlob`](../src/lib/circular-crop-to-blob.ts), 512×512) → `users.avatarThumb`. В списках и миниатюрах UI берёт URL из [`userAvatarListUrl`](../src/lib/user-avatar-display.ts) (`avatarThumb` с fallback на `avatar`). Рендер **внутри** `DialogContent`, плюс `showCloseButton={!cropOpen}` и `onOpenChange`, чтобы не закрыть анкету при открытой обрезке.

Страница «Мой профиль» ([`dashboard/profile/page.tsx`](../src/app/dashboard/profile/page.tsx), форма [`UserForm`](../src/components/admin/user-form.tsx) с `isProfilePage`): телефон из Firestore нормализуется через [`phoneFormValueFromStored`](../src/lib/phone-utils.ts) и показывается в [`PhoneInput`](../src/components/ui/phone-input.tsx) с той же маской, что при регистрации; аватар — круг + [`RegisterAvatarCropOverlay`](../src/components/auth/register-avatar-crop-overlay.tsx) с `variant="compact"` и [`scopeWithinElement`](../src/contexts/dashboard-main-column-scope.tsx) (размытие и модалка только над правой колонкой [`DashboardMainAndChatRail`](../src/components/dashboard/DashboardMainAndChatRail.tsx), список чатов не затемняется). 5. Google / Apple / Telegram / Яндекс: критерий «полного» профиля для дашборда — [`isRegistrationProfileComplete`](../src/lib/registration-profile-complete.ts) (имя, логин, email; **телефон не обязателен**). После входа [`finalizeOAuthCredential`](../src/hooks/use-auth.tsx) создаёт или **дозаполняет** `users/{uid}` (в т.ч. если документ уже создал trigger `onUserCreated`), подставляет уникальный `username`, телефон из провайдера только при отсутствии конфликта в `registrationIndex`, аватар из `photoURL` если в Firestore пусто или dicebear-заглушка. Форма без пароля — [`completeGoogleProfile`](../src/hooks/use-auth.tsx) и UI-блок [`RegisterDialogFormBlock`](../src/components/auth/register-dialog-form-block.tsx); редирект в [`dashboard/layout`](../src/app/dashboard/layout.tsx) только при полном профиле. Проверки `registrationIndex` с `{ exceptUid }`, чтобы не конфликтовать с собственной записью пользователя. После `setDoc`/`updateDoc` профиля `register` и `completeGoogleProfile` делают `getDoc` и [`setAppUser` в use-auth](../src/hooks/use-auth.tsx), плюс синхронный `appUserRef`. Слушатель `onSnapshot(users/{uid})` **игнорирует снимки из локального кэша** (`metadata.fromCache`), если они откатывают уже полный профиль к неполному — иначе после сохранения анкеты кэш присылал старый документ и снова открывал форму. Если кэш отдаёт **неполный** профиль при входе через Google, перед применением выполняется `getDocFromServer`: при полном документе на сервере подставляется он, чтобы не мигала анкета дозаполнения. Режим модалки (Google без пароля / email с паролем) задаётся флагом `googleProfileCompletionFlow` в контексте. Сразу после `signInWithPopup` / `getRedirectResult` / custom token [`finalizeOAuthCredential`](../src/hooks/use-auth.tsx) оборачивает первый `getDoc`/`setDoc`/`updateDoc` на `users/{uid}` в повторные попытки: `reload` + `getIdToken(true)`, чередование `getDocFromServer` / `getDoc` и пауза при `permission-denied` — стабилизует связку Auth ↔ persistent local cache Firestore.

## 3) Отправка сообщений

1. Клиент пишет сообщение в `conversations/{id}/messages/{messageId}`.
2. Обновляет parent `conversations/{id}` (последнее сообщение/счётчики/реакции).
3. Firestore trigger `onmessagecreated` при активном `disappearingMessageTtlSec` сначала выставляет на документе сообщения `expireAt` (Admin) для TTL/scheduled удаления; служебные сообщения (`__system__`, `systemEvent`) пропускаются. После этого trigger отправляет FCM другим участникам.
4. Для тредов trigger `onthreadmessagecreated` зеркалит логику `expireAt` для комментариев в `messages/{parentId}/thread/{id}`, затем шлёт отдельные уведомления.
5. Триггеры `onchatmessagemediatranscode` / `onchatthreadmessagemediatranscode` нормализуют неподдерживаемые форматы вложений (`webm/mov/...`) в `mp4/m4a`, обновляют `attachments` и поле `mediaNorm` (`pending|done|failed`, `failedIndexes`) в том же документе сообщения.
6. UI web/mobile показывает статус `mediaNorm`; при `failed` клиент может вызвать callable `retryChatMediaTranscode` для повторной обработки.
7. **Черновик сообщения (web):** неотправленный HTML ввода и контекст «ответ» сохраняются в `localStorage` per user (`src/lib/chat-message-draft-storage.ts`) при уходе с чата (`pagehide`, размонтирование поля ввода, смена `conversationId` / ключа треда). Ключ основного чата — `conversation.id`; для треда — `t:{conversationId}:{parentMessageId}`. После успешной отправки или правки сообщения запись очищается. В списке бесед [`ConversationItem`](../src/components/chat/ConversationItem.tsx) показывает метку «Черновик» и превью ([`useChatMainDraftPreview`](../src/hooks/use-chat-main-draft-preview.ts)). Вложения файлов в черновик не сериализуются.

   **Мобильный клиент (Flutter):** тот же префикс ключа `lighchat:chatDrafts:v1:` в SharedPreferences — [`mobile/app/lib/features/chat/data/chat_message_draft_storage.dart`](../../mobile/app/lib/features/chat/data/chat_message_draft_storage.dart); сохранение с debounce и снимок при смене беседы / `dispose` — [`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart); строка «Черновик · …» в списке — [`chat_list_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_list_screen.dart). Пока только основной чат (`conversationId`), без ключа треда `t:…`. Локальные файлы в хранилище не сериализуются.

   **Вставка/Drop контента (mobile):** композер чата в [`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart) поддерживает вставку из системного буфера (`SystemClipboard`) и внешнее drag&drop в область чата (`DropRegion`): plain text вставляется в текущую позицию курсора, файлы (изображения/видео/аудио/PDF) добавляются в `pendingAttachments` как при выборе через меню скрепки. При drag over показывается центральный индикатор drop-zone.
   **Композер и лента (mobile):** в [`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart) и [`chat_message_list.dart`](../../mobile/app/lib/features/chat/ui/chat_message_list.dart) список сообщений закреплён к нижней кромке (первое сообщение в пустом чате показывается снизу; новые сообщения выталкивают историю вверх). Левая кнопка композера — `+` с новым grid-меню вложений ([`composer_attachment_menu.dart`](../../mobile/app/lib/features/chat/ui/composer_attachment_menu.dart)); правая кнопка динамическая: по умолчанию микрофон, при наличии текста переключается на отправку.
   **Unread reset по viewport (mobile):** список сообщений ([`chat_message_list.dart`](../../mobile/app/lib/features/chat/ui/chat_message_list.dart)) публикует событие видимости строки (`visibleFraction >= 0.12`), а экраны [`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart) и [`thread_screen.dart`](../../mobile/app/lib/features/chat/ui/thread_screen.dart) вызывают read-mark API репозитория (`markMessagesAsRead` / `markManyMessagesAsRead`) с session-dedupe, чтобы счётчики `unreadCounts`/`unreadThreadCounts` сбрасывались по фактической видимости входящих сообщений (web parity).
   **Разделитель «Непрочитанные сообщения» (mobile):** id строки перед разделителем фиксируется на сессию открытия экрана (`_sessionUnreadSeparatorAnchorMessageId` в `chat_screen.dart` / `thread_screen.dart`): при первом появлении входящих непрочитанных в загруженном списке берётся тот же критерий, что и для «самого старого» непрочитанного; при прокрутке и прочитке через viewport id **не** пересчитывается (нет визуального «прыжка» линии), при появлении новых сообщений внизу позиция разделителя не смещается. Сброс в `null` при отсутствии входящих непрочитанных в загруженном окне и при смене беседы / корня треда (`didUpdateWidget`), чтобы после выхода и повторного входа якорь снова соответствовал актуальному oldest-unread. На web аналогичная семантика уже в `unreadSeparatorId` (`ChatWindow` / `ThreadWindow`).
   **Якорь прокрутки (mobile):** web-like плавающая кнопка внизу справа ([`chat_scroll_anchor_button.dart`](../../mobile/app/lib/features/chat/ui/chat_scroll_anchor_button.dart)) работает в двух режимах:
   1) `reaction mode` (приоритетный) — при новой реакции другого пользователя показывает emoji вместо стрелки и ведёт к сообщению/ветке реакции; после навигации пишет `lastReactionSeenAt.{uid}`.
   2) `unread mode` — если есть непрочитанные: первый тап к разделителю `Непрочитанные сообщения`, второй — в самый низ + массовый read-mark; без непрочитанных тап всегда скроллит в низ.
   При подгрузке истории у верхней границы, если новых сообщений нет, `chat_screen.dart` восстанавливает viewport и фиксирует позицию у старейшего края без «отпружинивания» вниз (iOS bounce-case).
   **Редактирование медиа перед отправкой (mobile):** в полосе pending-вложений ([`composer_pending_attachments_strip.dart`](../../mobile/app/lib/features/chat/ui/composer_pending_attachments_strip.dart)) кнопка edit доступна для изображений и видео. Для изображений открывается [`chat_image_editor_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_image_editor_screen.dart) (draw/crop/rotate/undo/reset + подпись, мульти-image flow). Для видео открывается [`chat_video_editor_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_video_editor_screen.dart) с раскадровкой на таймлайне (кадры + trim-handle’ы + draggable playhead), rotate/mute, отдельным crop-режимом (перемещение рамки и resize по углам/сторонам), freehand draw-слоем и экспортом через FFmpeg в новый локальный `.mp4` (включая overlay для нарисованных штрихов). Результат возвращает обновлённые вложения + caption; `chat_screen.dart` применяет caption в composer и отправляет стандартным upload-пайплайном.
   **Fullscreen video viewer (mobile):** в [`chat_media_viewer_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_media_viewer_screen.dart) для видео в полноэкранной галерее доступен расширенный плеер: timeline с текущим/общим временем, seek по слайдеру, шаги `±10s`, mute/unmute, выбор скорости (`1x/1.25x/1.5x/1.75x/2x`), переключение качества (`Auto/1080p/720p/480p`), действия «Сохранить» и «Показать в чате». Viewer работает в mixed-source режиме (`http(s)` и `file://`): локальные E2EE-файлы открываются через `VideoPlayerController.file`, backdrop у фото поддерживает `FileImage`, сохранение в галерею копирует локальный файл без `http.get`. Во время воспроизведения видео верхняя/нижняя chrome-панели скрываются; при показе шапки верхний ряд video-controls получает динамический inset по фактической высоте шапки, чтобы не перекрываться. ### Сквозное шифрование (E2E), опционально

- Протокол: единственная поддерживаемая версия — **v2 `v2-p256-aesgcm-multi`** (P-256 ECDH + AES-256-GCM + HKDF-SHA-256 + structured AAD, multi-device). Описание — [`07-e2ee-v2-protocol.md`](./07-e2ee-v2-protocol.md). Legacy v1 удалён в Phase 10 cleanup: `src/lib/e2ee/{enable-conversation,session-firestore,device-identity}.ts`, `src/lib/e2ee/v2/protocol-select.ts` и парное `readLegacyV1Device` на мобайле больше не существуют. Клиент, встретивший envelope / session с `protocolVersion != 'v2-p256-aesgcm-multi'`, помечает его `E2EE_UNSUPPORTED_PROTO` и триггерит self-heal (перекатывает эпоху в v2).
- Ключи устройств: web — IndexedDB (`identity-v2` store); mobile — `flutter_secure_storage`. Публичные ключи per-device: `users/{uid}/e2eeDevices/{deviceId}`.
- Включение в чате: [`enableE2eeOnConversationV2`](../src/lib/e2ee/v2/enable-conversation-v2.ts) создаёт `conversations/{id}/e2eeSessions/{epoch}` в v2-формате (вложенная мапа `wraps[uid][deviceId]`) и выставляет флаги на `Conversation`.
- Сообщения с `e2ee` хранят ciphertext; `text` с секретом не используется. Расшифровка только в клиенте ([`useE2eeConversation`](../src/hooks/use-e2ee-conversation.ts), [`ChatWindow`](../src/components/chat/ChatWindow.tsx), [`ThreadWindow`](../src/components/chat/ThreadWindow.tsx)) через `encryptOutgoingHtmlV2({ messageId })` / `decryptMessagePayload`.
- Гибрид: старые сообщения без `e2ee` остаются plaintext; новые после включения — только ciphertext.
- Авто-E2E при создании личного чата: [`autoEnableE2eeForNewDirectChat`](../src/lib/e2ee/v2/rollout.ts) читает `platformSettings/main.e2eeProtocolVersion` и вызывает [`tryAutoEnableE2eeV2NewDirectChat`](../src/lib/e2ee/v2/enable-conversation-v2.ts).
- Ручное включение/выключение для **конкретного личного** чата: шторка профиля [`ChatParticipantProfile`](../src/components/chat/ChatParticipantProfile.tsx) → пункт «Шифрование» → [`ConversationEncryptionPanel`](../src/components/chat/conversation-pages/ConversationEncryptionPanel.tsx); снятие флагов — [`disableE2eeOnConversation`](../src/lib/e2ee/disable-conversation-e2ee.ts). История с полем `e2ee` по-прежнему расшифровывается по эпохе в сообщении, даже если у чата `e2eeEnabled: false`.
- Пересылка сообщений с `e2ee` отключена ([`forward/page.tsx`](../src/app/dashboard/chat/forward/page.tsx)). Геолокация и опросы в E2E-чате блокируются на клиенте до отдельной доработки.
- **Mobile E2EE runtime:** [`MobileE2eeRuntime`](../mobile/app/lib/features/chat/data/e2ee_runtime.dart) (encrypt/decrypt текста + media). После Phase 10 cleanup любой не-v2 envelope возвращает `E2EE_UNSUPPORTED_PROTO`; UI показывает placeholder и при первом подходящем вызове провоцирует self-heal на чате. Исключение участника из E2EE-группы создаёт новый v2 `e2eeSessions/{epoch}` в [`GroupChatParticipantsManageView.tsx`](../src/components/chat/GroupChatParticipantsManageView.tsx).
- **RFC:** проектный RFC — [`07-e2ee-v2-protocol.md`](./07-e2ee-v2-protocol.md), тестовые векторы — [`e2ee-v2-test-vectors.json`](./e2ee-v2-test-vectors.json).

- **Phase 7 — media encryption:**
  - Ядро протокола: web — [`src/lib/e2ee/v2/media-crypto-v2.ts`](../src/lib/e2ee/v2/media-crypto-v2.ts) + [`media-upload-v2.ts`](../src/lib/e2ee/v2/media-upload-v2.ts); mobile — [`mobile/packages/lighchat_firebase/lib/src/e2ee/media_crypto.dart`](../mobile/packages/lighchat_firebase/lib/src/e2ee/media_crypto.dart) + [`media_storage.dart`](../mobile/packages/lighchat_firebase/lib/src/e2ee/media_storage.dart). На mobile тот же API предоставляет [`MobileE2eeRuntime.encryptMediaForSend` / `decryptMediaForView`](../mobile/app/lib/features/chat/data/e2ee_runtime.dart). Текст сообщения по‑прежнему идёт через `encryptOutgoingHtmlV2` — envelope'ы из `message.e2ee.attachments[]` живут параллельно обычному `attachments[]` (последний для стикеров/GIF остаётся plaintext).

Encryptable идут через `encryptAndUploadForSend` → `e2ee.attachments[]`. Для media-only сообщений в `e2ee.ciphertext` кодируется пустая строка — так [`onMessageCreated`](../functions/src/triggers/firestore/onMessageCreated.ts) корректно обрезает push-нотификацию до «Зашифрованное сообщение». Edit-path (`handleUpdateMessage`) перешифровывает только текст; existing `e2ee.attachments` сохраняются as-is (ciphertext привязан к `messageId`, re-upload при редактировании невозможен). Render — [`useE2eeHydratedMessages`](../src/hooks/use-e2ee-hydrated-messages.ts): хук ленивой подгрузки blob-URL'ов, подмешивает расшифрованные `ChatAttachment` в `message.attachments` перед рендером.
  - **UI send integration (mobile)**: [`e2ee_attachment_send_helper.dart`](../mobile/app/lib/features/chat/data/e2ee_attachment_send_helper.dart) — `prepareE2eeAttachmentsForSend` + `mergeE2eeEnvelopeWithMedia`. Call-sites в [`chat_screen.dart`](../mobile/app/lib/features/chat/ui/chat_screen.dart): `_submitComposer` (не-альбомные вложения), `_sendVideoCircleFile`, `_sendVoiceMessage`, `_sendStickerOrGifAttachment` (plaintext + пустой text-envelope). Альбом — [`outgoing_pending_media_album.dart`](../mobile/app/lib/features/chat/ui/outgoing_pending_media_album.dart) с `OutgoingAlbumE2eeContext`. `ChatRepository.sendTextMessage` / `sendThreadReply` принимают `e2eeEnvelope` + `messageIdOverride`; guard `_isE2eeActive && !hasE2ee` сохраняется чтобы исключить утечку plaintext при забытой интеграции. Для E2EE-active чатов mobile принудительно формирует envelope для любого исходящего (даже если global/override privacy-policy выключила `text/media`), чтобы не проваливаться в plaintext-block и не терять сообщения. Caption/текст шифруется всегда (в т.ч. пустая строка) — это гарантирует непустые iv/ct в `ChatMessageE2eePayload.fromJson`.

Бинарник пишется в persistent-кэш `applicationSupport/e2ee_cache/media/{conversationId}/{messageId}-{fileId}.{ext}`, `ChatAttachment.url` подставляется как `file://…`. `ChatCachedNetworkImage` и media-viewer поддерживают mixed-source рендер (`file://` + `http(s)`), поэтому E2EE-изображения отображаются в ленте и fullscreen как обычные. Для ошибок media-decrypt добавляется диагностический placeholder-attachment (`application/x-lighchat-e2ee-media-error`) и telemetry `media.decrypt.failure`, чтобы сбой был видим в UI. Галереи (`ChatScreen`, `ThreadScreen`, `conversation_media_links_files_screen.dart`) собираются из hydrated-списка, поэтому расшифрованные вложения попадают в «Медиа, ссылки и файлы» и fullscreen; парсер `ChatMessageE2eePayload` принимает media-only envelope'ы даже при отсутствующих `iv/ciphertext` (legacy mobile-only payload). Для video/circle превью в профиле чата first-frame cache нормализует `file://` input к локальному пути FFmpeg. Поле `ChatMessageE2eePayload.attachmentsJson` добавлено в [`lighchat_models`](../mobile/packages/lighchat_models/lib/lighchat_models.dart) и хранит сырой JSON (парсинг в `MediaEnvelopeV2` делает слой runtime — модели не зависят от `lighchat_firebase`).
  - **Secret Chat hard media views (server-enforced, online-only):** для секретных чатов Storage-read по пути `chat-attachments-enc/{cid}/{mid}/{fileId}/…` разрешён только при наличии короткоживущего grant-документа `conversations/{cid}/secretMediaKeyGrants/{uid}__{mid}__{fileId}` (см. `storage.rules`). На mobile `E2eeMessagesResolver` перед скачиванием вызывает `requestSecretMediaView`, ждёт fulfil → grant, затем выполняет download+decrypt и завершает `consumeSecretMediaKeyGrant`. Issuer-устройства параллельно слушают `secretMediaViewRequests` и вызывают `fulfillSecretMediaViewRequest` best-effort. Ограничение: если у второй стороны нет online устройства, fulfil/grant не будет выдан, и просмотр не состоится.
  - **Streaming-decrypt (Gap #2, minimal):** `useE2eeMediaAttachments.resolveStreamForView` поверх [`decryptMediaFileStreamV2`](../src/lib/e2ee/v2/media-crypto-v2.ts). Чанки (4 МиБ) скачиваются и расшифровываются по одному, склейка в `Blob`-URL производится в конце. Пик памяти ≈ 1 чанк поверх итогового plaintext (вместо 2× как у `resolveForView`); скачивание одного чанка идёт параллельно с расшифровкой предыдущего. Playback всё ещё стартует после полной загрузки; полноценный instant-playback (MSE / Service Worker) — отдельный итератив.

  - **Post-launch fix (web `deriveBits` usage)**: v2 unwrap использует `crypto.subtle.deriveBits(...)` для ECDH (см. [`webcrypto-v2.ts`](../src/lib/e2ee/v2/webcrypto-v2.ts)), но прежняя реализация [`importEcdhPrivateFromPkcs8`/`generateEcdhKeyPair`](../src/lib/e2ee/webcrypto.ts) импортировала приватник только с usages `['deriveKey']`. WebCrypto проверяет разрешение отдельно и возвращал `InvalidAccessError: key.usages does not permit this operation` / `CryptoKey doesn't support bits derivation` на unwrap v2-сессии. Симптомы: web-клиент не мог расшифровать сообщения из mobile (v2 сессия), а также получал `E2EE_NO_CHAT_KEY` на отправку, т.к. собственный `getAesKeyWithHealing` тоже падал. Фикс: добавить `'deriveBits'` в usages — `['deriveKey', 'deriveBits']`. Обратной совместимости это не ломает: pkcs8-байты в IndexedDB те же, usages назначаются при каждом `importKey`.
  - **Post-launch self-heal (multi-device sync)**: когда новое устройство появляется в `users/{uid}/e2eeDevices` после создания эпохи, без ротации оно не покрыто `wraps` и не может читать новые сообщения. Аналогично — после очистки кэша браузера новый публичник устройства не совпадает со старыми wrap'ами. Оба кейса теперь закрываются ленивой ротацией: web — [`healSessionForCurrentDevicesV2`](../src/lib/e2ee/v2/heal-session.ts), вызывается из [`useE2eeConversation`](../src/hooks/use-e2ee-conversation.ts) при mount чата и перед каждым encrypt; mobile — [`healSessionForCurrentDevices`](../mobile/packages/lighchat_firebase/lib/src/e2ee/heal_session.dart), вызывается из [`MobileE2eeRuntime._getOrFetchChatKey`](../mobile/app/lib/features/chat/data/e2ee_runtime.dart) на send-пути. Heal идёт через Firestore-транзакцию на `conversations/{id}.e2eeKeyEpoch` (защита от гонки между клиентами) и создаёт новый `e2eeSessions/{epoch}` с wraps на все активные устройства всех участников. Если heal вернул `healed:false` из-за конфликтной ротации другого клиента, send-путь на web и mobile дополнительно перечитывает актуальный `e2eeKeyEpoch` из `conversations/{id}` и шифрует уже под свежей эпохой (без повторных попыток в устаревшую epoch). Для web unwrap-ошибок вида `E2EE_NO_WRAP_FOR_DEVICE|USER` введено подавление повторов на уровне `(conversation, epoch)`, чтобы логи и UI не засыпались сотнями одинаковых сообщений до следующего heal/смены эпохи. Decrypt-путь heal **не вызывает**: старые сообщения остаются зашифрованы под ключом своей эпохи, и новая ротация их не восстановит. Это сохраняет семантику forward‑secrecy и при этом делает multi-device работоспособным без ручного «перевключите шифрование».

  - Новое поле `ChatMessage.systemEvent` (см. [`src/lib/types.ts`](../src/lib/types.ts) и [`mobile/packages/lighchat_models/lib/lighchat_models.dart`](../mobile/packages/lighchat_models/lib/lighchat_models.dart)). `senderId === '__system__'` + `systemEvent: { type, data }`, текст сообщения пуст. Не шифруется: дивайдер должен быть виден любому клиенту, включая новое устройство без ключа эпохи.
  - Публикация событий: web — [`postChatSystemEventV2` + `chatSystemEvents`](../src/lib/e2ee/v2/system-events.ts); mobile — [`postChatSystemEventV2` + `ChatSystemEventFactories`](../mobile/packages/lighchat_firebase/lib/src/e2ee/system_events.dart). Хук встроен в `enableE2eeOnConversationV2` и `revokeDeviceAndRekeyV2` (web) / `tryAutoEnableE2eeNewDirectChatMobile` и `revokeDeviceAndRekeyMobile` (mobile). Ошибка записи маркера не валит основное E2EE‑действие — divider считается UX‑украшением.
  - Рендер дивайдера: web — [`ChatSystemEventDivider`](../src/components/chat/ChatSystemEventDivider.tsx), встроен в `ChatWindow`/`ThreadWindow` через проверку `message.systemEvent && senderId === '__system__'`. Mobile — [`ChatSystemEventDivider`](../mobile/app/lib/features/chat/ui/chat_system_event_divider.dart) в `ChatMessageList`.
  - Unread semantics: `systemEvent`-сообщения (`senderId='__system__'`) исключены из подсчёта непрочитанных, не создают unread-separator и не участвуют в `markMessagesAsRead` на viewport/anchor.
  - Global read receipts privacy: флаг `users/{uid}.privacySettings.showReadReceipts=false` отключает проставление `messages.readAt` и `thread.readAt` на web/mobile (viewport + anchor). При этом сброс unread сохраняется: conversation/thread счётчики обнуляются через `markConversationAsRead` / `markThreadMessagesSeenWithoutReadReceipt` (web) и `markConversationAsRead` / `markThreadMessagesSeenWithoutReceipt` (mobile), чтобы список чатов не зависал в «непрочитанных».
  - Отпечаток E2EE собеседника в DM: web — [`E2eeFingerprintBadge`](../src/components/chat/E2eeFingerprintBadge.tsx), подключён в [`ConversationEncryptionPanel`](../src/components/chat/conversation-pages/ConversationEncryptionPanel.tsx). Mobile — [`E2eeFingerprintBadge`](../mobile/app/lib/features/chat/ui/e2ee_fingerprint_badge.dart) в [`ChatPartnerProfileSheet`](../mobile/app/lib/features/chat/ui/chat_partner_profile_sheet.dart). Показывается только когда `e2eeEnabled && !isGroup`.

## 4) 1:1 звонки

1. Создаётся `calls/{callId}` с caller/receiver и offer.
2. Trigger `oncallcreated` обновляет `userCalls` и отправляет входящий push: FCM (`users.fcmTokens`) + APNs VoIP (`users.voipTokens` для iOS).
3. Клиенты обмениваются offer/answer/candidates через `calls/{id}` и `calls/{id}/candidates`.
4. Завершение звонка обновляет статус и timestamps в документе call.
5. Mobile Flutter: из шапки личного чата ([`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart)) доступны кнопки аудио и видео; открываются отдельные экраны [`ChatAudioCallScreen`](../../mobile/app/lib/features/chat/ui/chat_audio_call_screen.dart) и [`ChatVideoCallScreen`](../../mobile/app/lib/features/chat/ui/chat_video_call_screen.dart). Для входящего звонка в открытом DM показывается баннер, который учитывает `calls.isVideo` и открывает нужный тип (`Принять`/`Отклонить`/`Открыть`).
6. Mobile/WebRTC flow: исходящий звонок создаёт `calls/{id}` со `status=calling` и `isVideo` (`false` для аудио, `true` для видео), затем публикует `offer`; принимающая сторона при подтверждении выставляет `answer` + `status=ongoing`; ICE-кандидаты пишутся в `calls/{id}/candidates` с `userId`. Завершение после активной фазы переводит звонок в `ended`; отклонение/отмена до соединения — в `cancelled`; непринятый звонок (таймаут 60с или отмена инициатором на дозвоне) — в `missed` (`endedAt` заполняется во всех terminal-кейсах).
7. Mobile call tones (web parity): [`ChatAudioCallScreen`](../../mobile/app/lib/features/chat/ui/chat_audio_call_screen.dart) и [`ChatVideoCallScreen`](../../mobile/app/lib/features/chat/ui/chat_video_call_screen.dart) используют [`ChatCallToneController`](../../mobile/app/lib/features/chat/data/chat_call_tones.dart) и storage-файлы `audio/ringtone.mp3` / `audio/ringback.mp3`. При входящем `status=calling` у получателя играет ringtone, при исходящем дозвоне у инициатора — ringback; при terminal-статусах (`ended/cancelled/missed`, legacy `rejected`) тоны останавливаются и перематываются в начало.
8. Mobile native incoming-call routing: push-слой [`PushNativeCallService`](../../mobile/app/lib/features/push/push_native_call_service.dart) обрабатывает `callId` из FCM и показывает системный incoming-call UI; action Accept открывает route [`/calls/incoming/:callId` в `app_router.dart`](../../mobile/app/lib/app_router.dart), где [`ChatIncomingCallEntryScreen`](../../mobile/app/lib/features/chat/ui/chat_incoming_call_entry_screen.dart) читает `calls/{id}` и переводит пользователя в нужный экран звонка (`audio`/`video`) с `existingCallId`.
9. iOS terminated/background parity: нативный PushKit delegate в [`ios/Runner/AppDelegate.swift`](../../mobile/app/ios/Runner/AppDelegate.swift) принимает VoIP push, маппит `callId` в deterministic UUID для CallKit и поднимает incoming UI через `SwiftFlutterCallkitIncomingPlugin`; accepted/declined действия продолжают обрабатываться Dart-слоем `PushNativeCallService`.
10. Mobile fallback без push: [`InAppIncomingCallScope`](../../mobile/app/lib/features/push/in_app_incoming_call_scope.dart) подписывается на `calls` для текущего `receiverId` и при `status in {'calling','ongoing'}` открывает route `/calls/incoming/:callId` даже без доставки APNs/FCM. Это покрывает сценарий, когда приложение уже открыто, но системные push недоступны.
11. Mobile call UX parity: входящий route [`ChatIncomingCallEntryScreen`](../../mobile/app/lib/features/chat/ui/chat_incoming_call_entry_screen.dart) резолвит имя/аватар через `users/*` + локальные `userContacts.contactProfiles`; terminal-переходы звонка пишут `endedBy`, а клиентский резолвер статуса учитывает роль viewer и `endedBy` (`missed/cancelled/ended`). Для аудиозвонка добавлен runtime-переключатель громкой связи.

## 5) Встречи (meetings)

1. Создаётся `meetings/{meetingId}`.
2. Перед входом открывается `JoinMeeting`; на публичной ссылке `/meetings/[meetingId]` поле имени показывается и для анонимных гостей (Firebase anonymous), чтобы имя задавалось до подключения WebRTC.
3. Участники входят в `participants`; trigger обновляет `userMeetings`.
4. ICE-настройки для meetings и 1:1 calls берутся из общего `src/lib/webrtc-ice-servers.ts`: клиент запрашивает `/api/webrtc/ice`, где сервер получает временные TURN-credentials (Metered) по приватным ENV `METERED_DOMAIN`/`METERED_API_KEY`; при ошибке используется STUN fallback.
5. Signaling: `meetings/{id}/signals`.
6. Встроенный чат: `meetings/{id}/messages`; опросы: `meetings/{id}/polls`.
7. Для приватных встреч запросы в `meetings/{id}/requests` через callable функции.

## 6) Админ и служебные потоки

- Callable admin endpoints (`createNewUser`, `updateUserAdmin`, backfill-операции).
- Периодический scheduler `checkUserPresence` чистит stale presence/meeting records и переводит `calls.status=calling` старше 60 секунд в `missed`.
- Server actions в `src/actions/*` выполняют read-heavy/privileged операции для UI-панелей.

## 7) iOS PWA performance guards

- Оболочка мобильного чата (`dashboard/layout.tsx`) задаёт высоту через `--lc-visual-vh` из [`useVisualViewportCssVars`](../src/hooks/use-visual-viewport-css-vars.ts): при клавиатуре берётся `visualViewport.height` + `offsetTop`, иначе `max(vv.height, window.innerHeight)` — после возврата PWA из фона WebKit иногда отстаёт по `visualViewport`, из‑за чего без `innerHeight` нижняя навигация оказывалась не у края экрана; дополнительно `visibilitychange` / `pageshow` и отложенные пересчёты синхронизируют высоту.
- Бейдж непрочитанных сообщений рассчитывается только в chat-маршрутах, чтобы не держать тяжёлые realtime-подписки на каждом экране dashboard.
- PWA onboarding выполняет длинные FCM-подписки в фоне и не блокирует UI длительным ожиданием.
- Overlay звонков подключается с небольшой отложенной инициализацией после входа в dashboard, чтобы разгрузить cold-start.
- В meetings ограничен объём realtime-данных на старте (лимиты истории чата и опросов).
- В meetings listeners для опросов/зала ожидания активируются по соответствующей вкладке sidebar, чтобы не держать лишние realtime-подписки постоянно.
- В chat list typing-listener на `conversations/{id}/typing` включается всегда для выбранного чата и только для видимых карточек при активной вкладке для остальных, чтобы вернуть индикатор "Печатает..." без постоянной нагрузки.
- В fullscreen media viewer чата на iOS touch-жесты разделены по зонам: media-зона отдаёт pinch/pan в `react-zoom-pan-pinch`, а swipe-down закрытия обрабатывается только из overlay-зоны.
- В ChatWindow/ThreadWindow догрузка истории вверх фиксирует `scrollTop/scrollHeight` перед увеличением window-limit и компенсирует смещение после рендера, чтобы избежать прыжков верстки при prepend старых сообщений.
- В ChatWindow/ThreadWindow, помимо `startReached`, включён fallback-триггер по `scrollTop <= 40px` с cooldown, чтобы догрузка истории стабильно срабатывала на устройствах, где виртуализатор не всегда эмитит верхнее событие.

## 8) Desktop (Electron): локальный UI и офлайн‑просмотр

1. **Собранное приложение** не грузит интерфейс с Firebase Hosting: в `electron/main.js` поднимается вложенный процесс Next **production** из **`.next-desktop/standalone/server.js`**. Сборка для десктопа идёт с `LIGHCHAT_DESKTOP_BUILD=1` (отдельный `distDir: '.next-desktop'` и `output: 'standalone'`), чтобы не делить `.next` с `next dev` и не ловить гонки/битые манифесты при трассировке; вшитый в `server.js` `distDir` указывает на **`./.next-desktop`** относительно каталога standalone, поэтому после `next build` скрипт `build:clean` копирует `.next-desktop/static` → **`standalone/.next-desktop/static`** (не в `.next/static`) и `public` → `standalone/public`.
2. Процесс стартует с `cwd` = каталог standalone и `ELECTRON_RUN_AS_NODE=1`, чтобы слой Node внутри сборки Electron выполнял `server.js` как обычный Node. Локальный URL — `http://localhost:3434` (вместо IP), чтобы OAuth/Firebase не упирался в отдельную авторизацию `127.0.0.1`. Для резолва модулей рантайм дополнительно прокидывает `NODE_PATH` с fallback-кандидатами (`standalone/node_modules`, `app.asar/node_modules`, `app.asar.unpacked/node_modules`) — это защищает запуск, если при переносе приложения отсутствует `standalone/node_modules`.
3. Окна `window.open` в Electron по умолчанию блокируются и уходят во внешний браузер, но для Google OAuth оставлен allowlist (`accounts.google.com`, `project-72b24.firebaseapp.com/__/auth/*`), чтобы `signInWithPopup` в desktop мог завершаться внутри приложения.
4. **Данные Firestore** при доступном IndexedDB уже кэшируются клиентским SDK (`persistentLocalCache` в [`src/firebase/index.ts`](../src/firebase/index.ts)); без сети доступно то, что было загружено ранее.
5. **Медиа из Storage** для Electron дополнительно кэшируются на диск в userData через IPC [`electron/media-cache.js`](../electron/media-cache.js), а в рендерер отдаются через кастомный протокол `lighchat-media://` (регистрация в [`electron/main.js`](../electron/main.js)). Это убирает ограничения загрузки `file://` из `http://localhost:*` и снижает CORS-проблемы для HEIC/HEIF цепочки [`useChatAttachmentDisplaySrc`](../src/components/chat/use-chat-attachment-display-src.ts) / [`src/lib/heic-heif-convert.ts`](../src/lib/heic-heif-convert.ts).
6. Упаковка: `electron-builder` включает `.next-desktop/standalone` и распаковывает его из asar (`asarUnpack`), чтобы нативные зависимости внутри standalone не ломались. Путь к серверу в рантайме резолвится относительно **`app.asar.unpacked`** (а не `app.asar`), иначе `server.js` не находится и окно остаётся пустым.
