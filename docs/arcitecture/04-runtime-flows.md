# 04: Runtime Flows

- 2026-05-12: **Модерация — hardening слой:** три самостоятельных улучшения по итогам обзора потока модерации.
  - **Rate-limit на `createMessageReportAction`** ([src/actions/moderation-actions.ts](../src/actions/moderation-actions.ts)): перед записью жалобы делается две `count()` aggregation query на `messageReports` по `reporterId` и `createdAt >= now - 1h` / `>= now - 24h`. Лимиты — 10/час и 30/сутки на uid. Защита от DoS на модерационную очередь и от спам-uploads в evidence-bucket. Fail-open при ошибке запроса (например missing index): логируем warning, легитимные жалобы не блокируем.
  - **Absolute TTL evidence** ([functions/src/triggers/scheduler/evidenceCleanupDaily.ts](../functions/src/triggers/scheduler/evidenceCleanupDaily.ts)): cron теперь делает два прохода. (1) Resolved TTL — 90 дней после `reviewedAt` для action_taken/dismissed, как раньше. (2) **Absolute TTL — 180 дней после `createdAt` для любого статуса**, включая pending. Защищает от «забытых» жалоб, у которых нет reviewedAt: расшифрованная копия E2EE-вложений не должна храниться бессрочно. В документе остаётся `evidenceClearedReason: 'resolved_ttl' | 'absolute_ttl'` для аудита.
  - **Sender notification** ([src/actions/moderation-actions.ts](../src/actions/moderation-actions.ts) → `notifySenderOfModerationAction`): автор скрытого сообщения / заблокированного аккаунта / получивший предупреждение теперь получает запись в `users/{senderId}/notifications/{auto}` с текстом «Ваше сообщение скрыто модератором / Ваш аккаунт заблокирован / Предупреждение от модератора», ссылкой на `/dashboard/support` и причиной (если задана). Пишется в `reviewAndHideReportAction` и в `reviewReportAction` (только при `status === 'action_taken'`). Раньше sender не получал никакого feedback'а — теперь есть точка входа для апелляции.
- 2026-05-12: **Модерация E2EE-сообщений — evidence-вложения:** для жалоб на сообщения с медиа в E2EE-чатах репортер опционально передаёт расшифрованную копию вложений в админ-only Storage. Flow: 1) [`ChatMessageItem`](../src/components/chat/ChatMessageItem.tsx) пробрасывает `message.e2ee.attachments` + `message.e2ee.epoch` + `getChatKeyRawV2ForEpoch` в [`ReportMessageDialog`](../src/components/chat/report-message-dialog.tsx). 2) В диалоге появляется чекбокс «Приложить вложения (N шт.)» с amber-disclosure про admin-only зону. 3) Перед сабмитом — [`uploadE2eeEvidence`](../src/lib/moderation/upload-e2ee-evidence.ts) расшифровывает каждый envelope через `downloadAndDecryptMediaFileV2` и кладёт plaintext в `moderation-evidence/{reporterUid}/{nonce}/attN.{ext}` (`nonce` = 24-char crypto-random). 4) `createMessageReportAction` валидирует, что каждый URL в `evidenceAttachments` указывает на `moderation-evidence/{reporterUid}/{nonce}/` — защита от подмены ссылок. 5) В Firestore документе `messageReports/{id}` появляются поля `evidenceAttachments: ChatAttachment[]` и `evidenceNonce: string`. **Storage rules** ([storage.rules](../storage.rules)): `moderation-evidence/{ownerUid}/**` — write только под своим uid + те же media-guards, read только админ, delete admin или owner (rollback). **Cleanup**: scheduled CF [`evidenceCleanupDaily`](../functions/src/triggers/scheduler/evidenceCleanupDaily.ts) (05:00 Europe/Moscow) удаляет GCS-объекты и зачищает поля для resolved-жалоб (status `action_taken`/`dismissed`) старше 90 дней после `reviewedAt`; сам документ репорта остаётся для аудита. Админ-панель ([admin-moderation-panel.tsx](../src/components/admin/admin-moderation-panel.tsx)) рендерит evidence отдельным amber-блоком с подписью «расшифрованная копия E2EE-вложений; будет удалена автоматически через 90 дней после resolved». E2EE-инвариант сознательно нарушается репортером — server по-прежнему не имеет ключей чата.
- 2026-05-12: **Модерация E2EE-сообщений — snapshot текста:** при сабмите жалобы из чата [`ChatMessageItem`](../src/components/chat/ChatMessageItem.tsx) теперь передаёт в [`ReportMessageDialog`](../src/components/chat/report-message-dialog.tsx) уже расшифрованный `displayTextHtml` (из `e2eeDecryptedByMessageId`), а не сырой `message.text` (для E2EE — ciphertext). UX в диалоге: для E2EE-сообщения показывается амбер-баннер «Сообщение зашифровано E2E. Чтобы модератор мог рассмотреть жалобу, мы приложим расшифрованную копию текста сообщения. Файлы (если есть) в этой версии пока не передаются — приложите скриншот в комментарии». Сервер ключей не имеет, без этой передачи модерация E2EE-чатов невозможна; нарушение E2EE-инварианта происходит сознательно репортером. Баннер админ-панели уточнён: при наличии snapshot'а — «Показан snapshot, отправленный автором жалобы; вложения недоступны», иначе «Snapshot текста автор жалобы не приложил». Полноценные evidence-аттачменты (re-upload расшифрованного медиа в отдельный admin-only bucket) — следующая итерация.

- 2026-05-12: **Админ-таб «Затраты» (BigQuery Billing Export):** новый таб в `/dashboard/admin` ([`AdminCostsPanel`](../src/components/admin/admin-costs-panel.tsx)) показывает реальные суммы по сервисам GCP за выбранный период. UI вызывает callable [`fetchBillingSummary`](../functions/src/triggers/http/fetchBillingSummary.ts) (region `us-central1`, admin-only), который читает таблицу Billing Export из BigQuery (`@google-cloud/bigquery` SDK) по конфигу `platformSettings/main.billing.{projectId, dataset, tableId}`. SQL — `SELECT service.description, SUM(cost), currency FROM <table> WHERE usage_start_time BETWEEN @from AND @to GROUP BY service, currency`. Identifiers валидируются регексом `[A-Za-z][A-Za-z0-9_-]*` перед подстановкой в backtick — защита от SQL-инъекции через имя таблицы. Если конфиг отсутствует — CF возвращает `{ ok:false, error:'not_configured' }`, UI открывает форму настройки. **Включение (один раз в GCP Console):** 1) Billing → Billing export → BigQuery (Standard usage cost) — Google создаёт таблицу `gcp_billing_export_v1_<billing-account-id>`. 2) В Cloud Functions SA добавить роли `roles/bigquery.dataViewer` на dataset и `roles/bigquery.jobUser` на project. 3) Заполнить projectId/dataset/tableId в админ-UI. Первая порция данных в BigQuery появляется в течение ~24 часов после включения.
- 2026-05-12: **Storage retention & quota enforcement:** ночные scheduled CF [`mediaRetentionCleanupDaily`](../functions/src/triggers/scheduler/mediaRetentionCleanupDaily.ts) (04:00 Europe/Moscow) и [`enforceStorageQuotasDaily`](../functions/src/triggers/scheduler/enforceStorageQuotasDaily.ts) (04:30) применяют `platformSettings/main.storage.{mediaRetentionDays, totalQuotaGb}` и поля `conversations/{cid}.storageQuotaBytes`. Гейтятся `enforcementMode`: `off` (default — no-op), `dry_run` (логирует, что бы удалили), `enforce` (реальное удаление). Общая логика — [`functions/src/lib/storage-quota-enforcement.ts`](../functions/src/lib/storage-quota-enforcement.ts): парсинг download-URL → `chat-attachments/*` или `chat-attachments-enc/*`, `evictMessageAttachments` стирает поле `attachments` и проставляет `mediaEvictedAt`/`mediaEvictedReason` (`retention | quota_conversation | quota_total`), `decrementConversationStorage` снижает счётчик `conversations.storage.totalBytes`. Per-conversation: oldest-first выселение, пока чат не упадёт ниже своей `storageQuotaBytes`. Global: после per-chat прохода считается суммарный `totalBytes` по всем чатам; если > `totalQuotaGb * GiB`, выселение идёт по «жирным» чатам в убывающем порядке. Per-user квота пока не enforce'ится (требуется отдельный счётчик usage). Админ-UI — [`AdminStorageSettingsPanel`](../src/components/admin/admin-storage-settings-panel.tsx): селектор режима, квоты пользователя/чата с поиском по имени/логину/email (новый компонент [`AdminQuotaTargetPicker`](../src/components/admin/admin-quota-target-picker.tsx)).
- 2026-05-09: **Mobile drag&drop стикеров (Telegram-style):** при drag&drop в [`ChatScreen`](../mobile/app/lib/features/chat/ui/chat_screen.dart)/[`ThreadScreen`](../mobile/app/lib/features/chat/ui/thread_screen.dart) каждый файл прогоняется через [`isLikelyStickerXFile`](../mobile/app/lib/features/chat/data/sticker_drop_heuristic.dart): PNG/WebP magic + декод через `package:image` + `width/height ≤ 512` + `numChannels ≥ 4` + четыре угла прозрачны (alpha < 16). Полный матч → дроп **не** идёт в `_pendingAttachments`, а сразу аплоадится через `uploadChatAttachmentFromXFile` и отправляется тем же путём, что стикеры из панели — `_sendStickerOrGifAttachment` / `_sendThreadStickerOrGifAttachment`. Это автоматически добавляет стикер в `RecentStickersStore`, сохраняет media-only E2EE‑envelope в зашифрованных чатах/тредах и не показывает caption-композер. Несовпадения (обычные фото/документы) идут в `_pendingAttachments` как раньше. Эвристика только для drop — paste из буфера, share intent и file picker сохраняют существующее поведение, иначе случайно превращали бы обычные PNG-скриншоты в стикер‑сообщения. Угловой пробинг — главный фильтр против false‑positive: у iOS «Lift Subject» углы 100% прозрачны, у скриншота — заполнены.

## 1) Auth и профиль

### QR-вход (Telegram-style привязка нового устройства)

- Экран входа [`/`](../src/app/page.tsx) рендерит **stage-stepper**: `entry` (Войти / Создать аккаунт) → `qr` (по умолчанию по «Войти») / `methods` (текущая форма email+OAuth).
- На stage `qr` рендерится [`QrLoginPanel`](../src/components/auth/QrLoginPanel.tsx): он берёт идентичность устройства из [`getOrCreateDeviceIdentityV2`](../src/lib/e2ee/v2/device-identity-v2.ts), вызывает callable [`requestQrLogin`](../functions/src/triggers/http/requestQrLogin.ts), кодирует ответ в QR (формат [`lighchat-login-v1`](../src/lib/qr-login/protocol.ts)) и подписывается на `qrLoginSessions/{sessionId}`. TTL — 90с с авторефрешем за 5с до истечения.
- Старое (залогиненное) устройство сканирует QR через [`QrScannerDialog`](../src/components/settings/QrScannerDialog.tsx) (web — [`@yudiel/react-qr-scanner`](https://www.npmjs.com/package/@yudiel/react-qr-scanner); mobile — переход к существующему [`/settings/e2ee-qr-pairing`](../mobile/app/lib/features/settings/ui/e2ee_qr_pairing_screen.dart) экрану). Парсер различает `lighchat-login-v1` и `v2-pairing-1` по полю `v` ([`parseQrLoginPayload`](../src/lib/qr-login/protocol.ts), Dart — [`parseQrLoginPayload`](../mobile/packages/lighchat_firebase/lib/src/e2ee/qr_login_protocol.dart)).
- После Allow на старом устройстве вызывается callable [`confirmQrLogin`](../functions/src/triggers/http/confirmQrLogin.ts): сервер сверяет `nonce` (хранится как SHA-256), создаёт Firebase custom token и кладёт его в документ. Новое устройство по листенеру забирает `customToken`, делает `signInWithCustomToken` и удаляет документ. Cleanup-функция [`cleanupQrLoginSessions`](../functions/src/triggers/scheduler/cleanupQrLoginSessions.ts) убирает мусор.
- **E2EE-handover (гибрид):** одновременно с подтверждением старое устройство запускает [`handoverDeviceAccessV2`](../src/lib/e2ee/v2/device-handover.ts) (Dart — [`handoverDeviceAccessMobile`](../mobile/packages/lighchat_firebase/lib/src/e2ee/device_link_handover.dart)). Алгоритм: (a) опубликовать `users/{uid}/e2eeDevices/{newDeviceId}` под `publicKeySpki` нового устройства; (b) для каждого E2EE-чата — re-wrap chatKey текущей эпохи под новый publicKey (добавляем wrap в session-doc) + создать новую эпоху с wraps под все устройства. Это даёт новому устройству и историю, и forward secrecy для будущих сообщений; собственный приватник старого устройства не передаётся.
- Пока новое устройство не получило свою wrap для активной эпохи (например, handover ещё бежит), [`useE2eeConversation`](../src/hooks/use-e2ee-conversation.ts) выставляет флаг `e2eeAwaitingWrap`; [`ChatWindow`](../src/components/chat/ChatWindow.tsx) показывает плашку «Синхронизация зашифрованного чата…» (i18n ключ `chat.encryptedSyncingBanner`).
- Управление устройствами вынесено на отдельную страницу [`/dashboard/settings/devices`](../src/app/dashboard/settings/devices/page.tsx); со страницы [`Privacy`](../src/app/dashboard/settings/privacy/page.tsx) убраны панели DevicesPanel и E2eeRecoveryPanel. ## 3) Отправка сообщений

1. Клиент пишет сообщение в `conversations/{id}/messages/{messageId}`.
2. Обновляет parent `conversations/{id}` (последнее сообщение/счётчики/реакции).
3. Firestore trigger `onmessagecreated` при активном `disappearingMessageTtlSec` сначала выставляет на документе сообщения `expireAt` (Admin) для TTL/scheduled удаления; служебные сообщения (`__system__`, `systemEvent`) пропускаются. После этого trigger отправляет FCM другим участникам.
4. Для тредов trigger `onthreadmessagecreated` зеркалит логику `expireAt` для комментариев в `messages/{parentId}/thread/{id}`, затем шлёт отдельные уведомления.
5. Триггеры `onchatmessagemediatranscode` / `onchatthreadmessagemediatranscode` нормализуют неподдерживаемые форматы вложений (`webm/mov/...`) в `mp4/m4a`, обновляют `attachments` и поле `mediaNorm` (`pending|done|failed`, `failedIndexes`) в том же документе сообщения.
6. UI web/mobile показывает статус `mediaNorm`; при `failed` клиент может вызвать callable `retryChatMediaTranscode` для повторной обработки.
7. **Черновик сообщения (web):** неотправленный HTML ввода и контекст «ответ» сохраняются в `localStorage` per user (`src/lib/chat-message-draft-storage.ts`) при уходе с чата (`pagehide`, размонтирование поля ввода, смена `conversationId` / ключа треда). Ключ основного чата — `conversation.id`; для треда — `t:{conversationId}:{parentMessageId}`. После успешной отправки или правки сообщения запись очищается. В списке бесед [`ConversationItem`](../src/components/chat/ConversationItem.tsx) показывает метку «Черновик» и превью ([`useChatMainDraftPreview`](../src/hooks/use-chat-main-draft-preview.ts)). Вложения файлов в черновик не сериализуются.

   **Мобильный клиент (Flutter):** тот же префикс ключа `lighchat:chatDrafts:v1:` в SharedPreferences — [`mobile/app/lib/features/chat/data/chat_message_draft_storage.dart`](../../mobile/app/lib/features/chat/data/chat_message_draft_storage.dart); сохранение с debounce и снимок при смене беседы / `dispose` — [`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart); строка «Черновик · …» в списке — [`chat_list_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_list_screen.dart). Пока только основной чат (`conversationId`), без ключа треда `t:…`. Локальные файлы в хранилище не сериализуются.

   **Вставка/Drop контента (mobile):** композер чата в [`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart) поддерживает вставку из системного буфера (`SystemClipboard`) и внешнее drag&drop в область чата (`DropRegion`): plain text вставляется в текущую позицию курсора, файлы (изображения/видео/аудио/PDF) добавляются в `pendingAttachments` как при выборе через меню скрепки. При drag over показывается центральный индикатор drop-zone.
   **Композер и лента (mobile):** в [`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart) и [`chat_message_list.dart`](../../mobile/app/lib/features/chat/ui/chat_message_list.dart) список сообщений закреплён к нижней кромке (первое сообщение в пустом чате показывается снизу; новые сообщения выталкивают историю вверх). Левая кнопка композера — `+` с новым grid-меню вложений ([`composer_attachment_menu.dart`](../../mobile/app/lib/features/chat/ui/composer_attachment_menu.dart)); правая кнопка динамическая: по умолчанию микрофон, при наличии текста переключается на отправку.
   **Unread reset по viewport (mobile):** список сообщений ([`chat_message_list.dart`](../../mobile/app/lib/features/chat/ui/chat_message_list.dart)) публикует событие видимости строки (`visibleFraction >= 0.12`), а экраны [`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart) и [`thread_screen.dart`](../../mobile/app/lib/features/chat/ui/thread_screen.dart) вызывают read-mark API репозитория (`markMessagesAsRead` / `markManyMessagesAsRead`) с session-dedupe, чтобы счётчики `unreadCounts`/`unreadThreadCounts` сбрасывались по фактической видимости входящих сообщений (web parity).
   **Разделитель «Непрочитанные сообщения» (mobile):** id строки перед разделителем фиксируется на сессию открытия экрана (`_sessionUnreadSeparatorAnchorMessageId` в `chat_screen.dart` / `thread_screen.dart`): при первом появлении входящих непрочитанных в загруженном списке берётся тот же критерий, что и для «самого старого» непрочитанного; при прокрутке и прочитке через viewport id **не** пересчитывается (нет визуального «прыжка» линии), при появлении новых сообщений внизу позиция разделителя не смещается. Сброс в `null` при отсутствии входящих непрочитанных в загруженном окне и при смене беседы / корня треда (`didUpdateWidget`), чтобы после выхода и повторного входа якорь снова соответствовал актуальному oldest-unread. На web аналогичная семантика уже в `unreadSeparatorId` (`ChatWindow` / `ThreadWindow`).
   **Якорь прокрутки (mobile):** web-like плавающая кнопка внизу справа ([`chat_scroll_anchor_button.dart`](../../mobile/app/lib/features/chat/ui/chat_scroll_anchor_button.dart)) работает в двух режимах:
   1) `reaction mode` (приоритетный) — при новой реакции другого пользователя показывает emoji вместо стрелки и ведёт к сообщению/ветке реакции; после навигации пишет `lastReactionSeenAt.{uid}`.
   2) `unread mode` — если есть непрочитанные: первый тап к разделителю `Непрочитанные сообщения`, второй — в самый низ + массовый read-mark; без непрочитанных тап всегда скроллит в низ.
   При подгрузке истории у верхней границы, если новых сообщений нет, `chat_screen.dart` восстанавливает viewport и фиксирует позицию у старейшего края без «отпружинивания» вниз (iOS bounce-case).
   **Редактирование медиа перед отправкой (mobile):** в полосе pending-вложений ([`composer_pending_attachments_strip.dart`](../../mobile/app/lib/features/chat/ui/composer_pending_attachments_strip.dart)) кнопка edit доступна для изображений и видео. Для изображений открывается [`chat_image_editor_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_image_editor_screen.dart) (draw/crop/rotate/undo/reset + подпись, мульти-image flow). Для видео открывается [`chat_video_editor_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_video_editor_screen.dart) с раскадровкой на таймлайне (кадры + trim-handle’ы + draggable playhead), rotate/mute, отдельным crop-режимом (перемещение рамки и resize по углам/сторонам), freehand draw-слоем и экспортом через FFmpeg в новый локальный `.mp4` (включая overlay для нарисованных штрихов). Результат возвращает обновлённые вложения + caption; `chat_screen.dart` применяет caption в composer и отправляет стандартным upload-пайплайном.
   **Fullscreen video viewer (mobile):** в [`chat_media_viewer_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_media_viewer_screen.dart) для видео в полноэкранной галерее доступен расширенный плеер: timeline с текущим/общим временем, seek по слайдеру, шаги `±10s`, mute/unmute, выбор скорости (`1x/1.25x/1.5x/1.75x/2x`), переключение качества (`Auto/1080p/720p/480p`), действия «Сохранить» и «Показать в чате». Во время воспроизведения видео верхняя/нижняя chrome-панели скрываются; при показе шапки верхний ряд video-controls получает динамический inset по фактической высоте шапки, чтобы не перекрываться. Для Android добавлен PiP через `MethodChannel` `lighchat/pip` (activity в [`MainActivity.kt`](../../mobile/app/android/app/src/main/kotlin/com/lighchat/lighchat_mobile/MainActivity.kt), `AndroidManifest` с `supportsPictureInPicture`).
   **Inline autoplay (mobile):** [`message_video_attachment.dart`](../../mobile/app/lib/features/chat/ui/message_video_attachment.dart) при ручной паузе пользователем ставит локальный флаг suppression — после скролла из/в viewport видео не автозапускается, пока пользователь снова явно не нажмёт Play.

В ленте воспроизводится стандартным [`MessageVoiceAttachment`](../../mobile/app/lib/features/chat/ui/message_voice_attachment.dart).

### Сквозное шифрование (E2E), опционально

- Протокол: единственная поддерживаемая версия — **v2 `v2-p256-aesgcm-multi`** (P-256 ECDH + AES-256-GCM + HKDF-SHA-256 + structured AAD, multi-device). Описание — [`07-e2ee-v2-protocol.md`](./07-e2ee-v2-protocol.md). Legacy v1 удалён в Phase 10 cleanup: `src/lib/e2ee/{enable-conversation,session-firestore,device-identity}.ts`, `src/lib/e2ee/v2/protocol-select.ts` и парное `readLegacyV1Device` на мобайле больше не существуют. Клиент, встретивший envelope / session с `protocolVersion != 'v2-p256-aesgcm-multi'`, помечает его `E2EE_UNSUPPORTED_PROTO` и триггерит self-heal (перекатывает эпоху в v2).
- Ключи устройств: web — IndexedDB (`identity-v2` store); mobile — `flutter_secure_storage`. Публичные ключи per-device: `users/{uid}/e2eeDevices/{deviceId}`.
- Включение в чате: [`enableE2eeOnConversationV2`](../src/lib/e2ee/v2/enable-conversation-v2.ts) создаёт `conversations/{id}/e2eeSessions/{epoch}` в v2-формате (вложенная мапа `wraps[uid][deviceId]`) и выставляет флаги на `Conversation`.
- Сообщения с `e2ee` хранят ciphertext; `text` с секретом не используется. Расшифровка только в клиенте ([`useE2eeConversation`](../src/hooks/use-e2ee-conversation.ts), [`ChatWindow`](../src/components/chat/ChatWindow.tsx), [`ThreadWindow`](../src/components/chat/ThreadWindow.tsx)) через `encryptOutgoingHtmlV2({ messageId })` / `decryptMessagePayload`.
- Гибрид: старые сообщения без `e2ee` остаются plaintext; новые после включения — только ciphertext.
- Авто-E2E при создании личного чата: [`autoEnableE2eeForNewDirectChat`](../src/lib/e2ee/v2/rollout.ts) читает `platformSettings/main.e2eeProtocolVersion` и вызывает [`tryAutoEnableE2eeV2NewDirectChat`](../src/lib/e2ee/v2/enable-conversation-v2.ts).
- Ручное включение/выключение для **конкретного личного** чата: шторка профиля [`ChatParticipantProfile`](../src/components/chat/ChatParticipantProfile.tsx) → пункт «Шифрование» → [`ConversationEncryptionPanel`](../src/components/chat/conversation-pages/ConversationEncryptionPanel.tsx); снятие флагов — [`disableE2eeOnConversation`](../src/lib/e2ee/disable-conversation-e2ee.ts). История с полем `e2ee` по-прежнему расшифровывается по эпохе в сообщении, даже если у чата `e2eeEnabled: false`.
- Пересылка сообщений с `e2ee` отключена ([`forward/page.tsx`](../src/app/dashboard/chat/forward/page.tsx)). Геолокация и опросы в E2E-чате блокируются на клиенте до отдельной доработки.
- **Mobile E2EE runtime:** [`MobileE2eeRuntime`](../mobile/app/lib/features/chat/data/e2ee_runtime.dart) (encrypt/decrypt текста + media). После Phase 10 cleanup любой не-v2 envelope возвращает `E2EE_UNSUPPORTED_PROTO`; UI показывает placeholder и при первом подходящем вызове провоцирует self-heal на чате. Исключение участника из E2EE-группы создаёт новый v2 `e2eeSessions/{epoch}` в [`GroupChatParticipantsManageView.tsx`](../src/components/chat/GroupChatParticipantsManageView.tsx).
- **RFC:** проектный RFC — [`07-e2ee-v2-protocol.md`](./07-e2ee-v2-protocol.md), тестовые векторы — [`e2ee-v2-test-vectors.json`](./e2ee-v2-test-vectors.json).

- **Phase 7 — media encryption:**
  - Ядро протокола: web — [`src/lib/e2ee/v2/media-crypto-v2.ts`](../src/lib/e2ee/v2/media-crypto-v2.ts) + [`media-upload-v2.ts`](../src/lib/e2ee/v2/media-upload-v2.ts); mobile — [`mobile/packages/lighchat_firebase/lib/src/e2ee/media_crypto.dart`](../mobile/packages/lighchat_firebase/lib/src/e2ee/media_crypto.dart) + [`media_storage.dart`](../mobile/packages/lighchat_firebase/lib/src/e2ee/media_storage.dart). Текст сообщения по‑прежнему идёт через `encryptOutgoingHtmlV2` — envelope'ы из `message.e2ee.attachments[]` живут параллельно обычному `attachments[]` (последний для стикеров/GIF остаётся plaintext).

Encryptable идут через `encryptAndUploadForSend` → `e2ee.attachments[]`. Для media-only сообщений в `e2ee.ciphertext` кодируется пустая строка — так [`onMessageCreated`](../functions/src/triggers/firestore/onMessageCreated.ts) корректно обрезает push-нотификацию до «Зашифрованное сообщение». Edit-path (`handleUpdateMessage`) перешифровывает только текст; existing `e2ee.attachments` сохраняются as-is (ciphertext привязан к `messageId`, re-upload при редактировании невозможен). Render — [`useE2eeHydratedMessages`](../src/hooks/use-e2ee-hydrated-messages.ts): хук ленивой подгрузки blob-URL'ов, подмешивает расшифрованные `ChatAttachment` в `message.attachments` перед рендером. `blob:`-URL для локальных превью и optimistic-сообщений теперь освобождаются явно после ack/ошибки и при unmount.
  - **UI send integration (mobile)**: [`e2ee_attachment_send_helper.dart`](../mobile/app/lib/features/chat/data/e2ee_attachment_send_helper.dart) — `prepareE2eeAttachmentsForSend` + `mergeE2eeEnvelopeWithMedia`. Call-sites в [`chat_screen.dart`](../mobile/app/lib/features/chat/ui/chat_screen.dart): `_submitComposer` (не-альбомные вложения), `_sendVideoCircleFile`, `_sendVoiceMessage`, `_sendStickerOrGifAttachment` (plaintext + пустой text-envelope). Для стабильности multi-media send в E2EE-чатах действует режим **stability-first**: encrypt/upload в альбоме идёт строго по одному файлу (без `Future.wait` по файлам), для encryptable-video применяется `maybeCompressVideoForSend720p` перед `readAsBytes/encrypt`, а батч проверяется ранними лимитами (`max files` + `max total bytes`) до старта шифрования. `ChatRepository.sendTextMessage` / `sendThreadReply` принимают `e2eeEnvelope` + `messageIdOverride`; guard `_isE2eeActive && !hasE2ee` сохраняется чтобы исключить утечку plaintext при забытой интеграции. Caption/текст шифруется всегда (в т.ч. пустая строка) — это гарантирует непустые iv/ct в `ChatMessageE2eePayload.fromJson`.

[`webcrypto-v2.ts`](../src/lib/e2ee/v2/webcrypto-v2.ts)), но прежняя реализация [`importEcdhPrivateFromPkcs8`/`generateEcdhKeyPair`](../src/lib/e2ee/webcrypto.ts) импортировала приватник только с usages `['deriveKey']`.   - **Post-launch self-heal (multi-device sync)**: когда новое устройство появляется в `users/{uid}/e2eeDevices` после создания эпохи, без ротации оно не покрыто `wraps` и не может читать новые сообщения. Аналогично — после очистки кэша браузера новый публичник устройства не совпадает со старыми wrap'ами. Оба кейса теперь закрываются ленивой ротацией: web — [`healSessionForCurrentDevicesV2`](../src/lib/e2ee/v2/heal-session.ts), вызывается из [`useE2eeConversation`](../src/hooks/use-e2ee-conversation.ts) при mount чата и перед каждым encrypt; mobile — [`healSessionForCurrentDevices`](../mobile/packages/lighchat_firebase/lib/src/e2ee/heal_session.dart), вызывается из [`MobileE2eeRuntime._getOrFetchChatKey`](../mobile/app/lib/features/chat/data/e2ee_runtime.dart) на send-пути. Heal идёт через Firestore-транзакцию на `conversations/{id}.e2eeKeyEpoch` (защита от гонки между клиентами) и создаёт новый `e2eeSessions/{epoch}` с wraps на все активные устройства всех участников. Если heal вернул `healed:false` из-за конфликтной ротации другого клиента, send-путь на web и mobile дополнительно перечитывает актуальный `e2eeKeyEpoch` из `conversations/{id}` и шифрует уже под свежей эпохой (без повторных попыток в устаревшую epoch). Для web unwrap-ошибок вида `E2EE_NO_WRAP_FOR_DEVICE|USER` введено подавление повторов на уровне `(conversation, epoch)`, чтобы логи и UI не засыпались сотнями одинаковых сообщений до следующего heal/смены эпохи. Decrypt-путь heal **не вызывает**: старые сообщения остаются зашифрованы под ключом своей эпохи, и новая ротация их не восстановит. Это сохраняет семантику forward‑secrecy и при этом делает multi-device работоспособным без ручного «перевключите шифрование».
- **Phase 8 — timeline markers + fingerprint UI:**
  - Новое поле `ChatMessage.systemEvent` (см. [`src/lib/types.ts`](../src/lib/types.ts) и [`mobile/packages/lighchat_models/lib/lighchat_models.dart`](../mobile/packages/lighchat_models/lib/lighchat_models.dart)). `senderId === '__system__'` + `systemEvent: { type, data }`, текст сообщения пуст. Не шифруется: дивайдер должен быть виден любому клиенту, включая новое устройство без ключа эпохи.
  - Публикация событий: web — [`postChatSystemEventV2` + `chatSystemEvents`](../src/lib/e2ee/v2/system-events.ts); mobile — [`postChatSystemEventV2` + `ChatSystemEventFactories`](../mobile/packages/lighchat_firebase/lib/src/e2ee/system_events.dart). Хук встроен в `enableE2eeOnConversationV2` и `revokeDeviceAndRekeyV2` (web) / `tryAutoEnableE2eeNewDirectChatMobile` и `revokeDeviceAndRekeyMobile` (mobile). Ошибка записи маркера не валит основное E2EE‑действие — divider считается UX‑украшением.
  - Рендер дивайдера: web — [`ChatSystemEventDivider`](../src/components/chat/ChatSystemEventDivider.tsx), встроен в `ChatWindow`/`ThreadWindow` через проверку `message.systemEvent && senderId === '__system__'`. Mobile — [`ChatSystemEventDivider`](../mobile/app/lib/features/chat/ui/chat_system_event_divider.dart) в `ChatMessageList`.
  - Unread semantics: `systemEvent`-сообщения (`senderId='__system__'`) исключены из подсчёта непрочитанных, не создают unread-separator и не участвуют в `markMessagesAsRead` на viewport/anchor.
  - Global read receipts privacy: флаг `users/{uid}.privacySettings.showReadReceipts=false` (или per-conversation `chatConversationPrefs.suppressReadReceipts=true`) отключает проставление публичного `messages.readAt`/`thread.readAt` (галочки прочтения собеседнику не показываются), но НЕ ломает локальный сброс unread: вместо `readAt` пишется персональная отметка `messages.readByUid.{viewerUid}` (или `thread.readByUid.{viewerUid}`) — её видит только сам пользователь. `markMessagesAsRead`/`markManyMessagesAsRead` (web — [`src/lib/chat-utils.ts`](../src/lib/chat-utils.ts), mobile — [`mobile/packages/lighchat_firebase/lib/src/chat_repository.dart`](../mobile/packages/lighchat_firebase/lib/src/chat_repository.dart)) принимают `skipReadReceipt`: при `true` пишут только `readByUid.{uid}`, при `false` — оба поля. `isIncomingUnreadForViewer` (web — [`src/lib/message-read-status.ts`](../src/lib/message-read-status.ts), mobile — встроен в `chat_screen.dart`/`thread_screen.dart`) трактует сообщение как unread, только если нет ни `readAt`, ни `readByUid[me]` — это исключает уход счётчика в минус при повторном открытии чата.
  - Отпечаток E2EE собеседника в DM: web — [`E2eeFingerprintBadge`](../src/components/chat/E2eeFingerprintBadge.tsx), подключён в [`ConversationEncryptionPanel`](../src/components/chat/conversation-pages/ConversationEncryptionPanel.tsx). Mobile — [`E2eeFingerprintBadge`](../mobile/app/lib/features/chat/ui/e2ee_fingerprint_badge.dart) в [`ChatPartnerProfileSheet`](../mobile/app/lib/features/chat/ui/chat_partner_profile_sheet.dart). Показывается только когда `e2eeEnabled && !isGroup`.

## 4) 1:1 звонки

1. Создаётся `calls/{callId}` с caller/receiver и offer.
2. Trigger `oncallcreated` обновляет `userCalls` и отправляет входящий push: FCM (`users.fcmTokens`) + APNs VoIP (`users.voipTokens` для iOS).
3. Клиенты обмениваются offer/answer/candidates через `calls/{id}` и `calls/{id}/candidates`.
4. Завершение звонка обновляет статус и timestamps в документе call.
5. Mobile Flutter: из шапки личного чата ([`chat_screen.dart`](../../mobile/app/lib/features/chat/ui/chat_screen.dart)) доступны кнопки аудио и видео; открываются отдельные экраны [`ChatAudioCallScreen`](../../mobile/app/lib/features/chat/ui/chat_audio_call_screen.dart) и [`ChatVideoCallScreen`](../../mobile/app/lib/features/chat/ui/chat_video_call_screen.dart). Для входящего звонка в открытом DM показывается баннер, который учитывает `calls.isVideo` и открывает нужный тип (`Принять`/`Отклонить`/`Открыть`).
6. Mobile/WebRTC flow: исходящий звонок создаёт `calls/{id}` со `status=calling` и `isVideo` (`false` для аудио, `true` для видео), затем публикует `offer`; принимающая сторона при подтверждении выставляет `answer` + `status=ongoing`; ICE-кандидаты пишутся в `calls/{id}/candidates` с `userId`. Завершение после активной фазы переводит звонок в `ended`; отклонение/отмена до соединения — в `cancelled`; непринятый звонок (таймаут 60с или отмена инициатором на дозвоне) — в `missed` (`endedAt` заполняется во всех terminal-кейсах).
7. Mobile call tones (web parity): [`ChatAudioCallScreen`](../../mobile/app/lib/features/chat/ui/chat_audio_call_screen.dart) и [`ChatVideoCallScreen`](../../mobile/app/lib/features/chat/ui/chat_video_call_screen.dart) используют [`ChatCallToneController`](../../mobile/app/lib/features/chat/data/chat_call_tones.dart) и storage-файлы `audio/ringtone.mp3` / `audio/ringback.mp3`. При входящем `status=calling` у получателя играет ringtone, при исходящем дозвоне у инициатора — ringback; при terminal-статусах (`ended/cancelled/missed`, legacy `rejected`) тоны останавливаются и перематываются в начало.
8. Mobile native incoming-call routing: push-слой [`PushNativeCallService`](../../mobile/app/lib/features/push/push_native_call_service.dart) обрабатывает `callId` из FCM и показывает системный incoming-call UI; action Accept открывает route [`/calls/incoming/:callId` в `app_router.dart`](../../mobile/app/lib/app_router.dart), где [`ChatIncomingCallEntryScreen`](../../mobile/app/lib/features/chat/ui/chat_incoming_call_entry_screen.dart) читает `calls/{id}` и переводит пользователя в нужный экран звонка (`audio`/`video`) с `existingCallId`.
9. iOS terminated/background parity: нативный PushKit delegate в [`ios/Runner/AppDelegate.swift`](../../mobile/app/ios/Runner/AppDelegate.swift) принимает VoIP push, маппит `callId` в deterministic UUID для CallKit и поднимает incoming UI через `SwiftFlutterCallkitIncomingPlugin`; accepted/declined действия продолжают обрабатываться Dart-слоем `PushNativeCallService`.
10. Mobile fallback без push: [`InAppIncomingCallScope`](../../mobile/app/lib/features/push/in_app_incoming_call_scope.dart) подписывается на `calls` для текущего `receiverId` и при `status in {'calling','ongoing'}` открывает route `/calls/incoming/:callId` даже без доставки APNs/FCM. Это покрывает сценарий, когда приложение уже открыто, но системные push недоступны.
11. Mobile call UX parity: входящий route [`ChatIncomingCallEntryScreen`](../../mobile/app/lib/features/chat/ui/chat_incoming_call_entry_screen.dart) резолвит имя/аватар через `users/*` + локальные `userContacts.contactProfiles`; terminal-переходы звонка пишут `endedBy`, а клиентский резолвер статуса учитывает роль viewer и `endedBy` (`missed/cancelled/ended`). Для аудиозвонка добавлен runtime-переключатель громкой связи.

## 5) Встречи (meetings)

1. Создаётся `meetings/{meetingId}`.
2. Перед входом открывается `JoinMeeting`; на публичной ссылке `/meetings/[meetingId]` поле имени показывается и для анонимных гостей (Firebase anonymous), чтобы имя задавалось до подключения WebRTC.
3. **Гостевой доступ (Anonymous Auth):**
   - В [`use-auth.tsx`](../src/hooks/use-auth.tsx) для `firebaseUser.isAnonymous` `isAuthenticated=false` и `isGuest=true`. Гость получает in-memory `user` (нужен `MeetingPage`), но dashboard layout считает его не-аутентифицированным и редиректит на `/`. Без этого ветка «incomplete profile» (`isRegistrationProfileComplete(user)` для гостя false) уводила гостя на `/dashboard/profile` и предлагала «дозаполнить профиль».
   - `firestore.rules` для `users/{uid}` запрещают `create`/`update` от `isAnonymousGuest()` (claim `request.auth.token.firebase.sign_in_provider == 'anonymous'`) — defense-in-depth, чтобы клиент не мог писать профиль из обхода UI. Документ `users/{uid}` для гостя продолжает создавать `onUserCreated` через Admin SDK.
   - Scheduled CF [`cleanupGuestAccounts`](../functions/src/triggers/scheduler/cleanupGuestAccounts.ts) раз в час удаляет анонимные Auth-записи старше 24 часов и `recursiveDelete` их `users/{uid}`/`userChats/{uid}`/`userContacts/{uid}`/`userCalls/{uid}`/`userMeetings/{uid}`. Фильтр гостя — `providerData.length === 0` и uid без префиксов `tg_`/`ya_` (custom-token бриджей).
4. Участники входят в `participants`; trigger обновляет `userMeetings`.
5. ICE-настройки для meetings и 1:1 calls берутся из общего `src/lib/webrtc-ice-servers.ts`: клиент запрашивает `/api/webrtc/ice`, где сервер получает временные TURN-credentials (Metered) по приватным ENV `METERED_DOMAIN`/`METERED_API_KEY`; при ошибке используется STUN fallback.
6. Signaling: `meetings/{id}/signals`.
7. Встроенный чат: `meetings/{id}/messages`; опросы: `meetings/{id}/polls`.
8. Для приватных встреч запросы в `meetings/{id}/requests` через callable функции.

## 6) Админ и служебные потоки

- Callable admin endpoints (`createNewUser`, `updateUserAdmin`, backfill-операции).
- Периодический scheduler `checkUserPresence` чистит stale presence/meeting records и переводит `calls.status=calling` старше 60 секунд в `missed`.
- Scheduled CF [`cleanupGuestAccounts`](../functions/src/triggers/scheduler/cleanupGuestAccounts.ts) раз в час удаляет гостевые (Anonymous Auth) аккаунты старше 24 часов и их Firestore-документы (см. §5 ›3 «Гостевой доступ»).
- Server actions в `src/actions/*` выполняют read-heavy/privileged операции для UI-панелей.

## 7) iOS PWA performance guards

- Оболочка мобильного чата (`dashboard/layout.tsx`) задаёт высоту через `--lc-visual-vh` из [`useVisualViewportCssVars`](../src/hooks/use-visual-viewport-css-vars.ts): при клавиатуре берётся `visualViewport.height` + `offsetTop`, иначе `max(vv.height, window.innerHeight)` — после возврата PWA из фона WebKit иногда отстаёт по `visualViewport`, из‑за чего без `innerHeight` нижняя навигация оказывалась не у края экрана; дополнительно `visibilitychange` / `pageshow` и отложенные пересчёты синхронизируют высоту.
- Бейдж непрочитанных сообщений рассчитывается только в chat-маршрутах, чтобы не держать тяжёлые realtime-подписки на каждом экране dashboard.
- PWA onboarding выполняет длинные FCM-подписки в фоне и не блокирует UI длительным ожиданием.
- Overlay звонков подключается с небольшой отложенной инициализацией после входа в dashboard, чтобы разгрузить cold-start.
- В meetings ограничен объём realtime-данных на старте (лимиты истории чата и опросов).
- В meetings listeners для опросов/зала ожидания активируются по соответствующей вкладке sidebar, чтобы не держать лишние realtime-подписки постоянно.
- В chat list typing-listener на `conversations/{id}/typing` включается всегда для выбранного чата и только для видимых карточек при активной вкладке для остальных, чтобы вернуть индикатор "Печатает..." без постоянной нагрузки.
- В fullscreen media viewer чата на iOS touch-жесты разделены по зонам: media-зона отдаёт pinch/pan в `react-zoom-pan-pinch`, а swipe-down закрытия обрабатывается только из overlay-зоны.
- В ChatWindow/ThreadWindow догрузка истории вверх фиксирует `scrollTop/scrollHeight` перед увеличением window-limit и компенсирует смещение после рендера, чтобы избежать прыжков верстки при prepend старых сообщений.
- В ChatWindow/ThreadWindow, помимо `startReached`, включён fallback-триггер по `scrollTop <= 40px` с cooldown, чтобы догрузка истории стабильно срабатывала на устройствах, где виртуализатор не всегда эмитит верхнее событие.

## 8) Desktop (Flutter): нативная macOS/Windows/Linux-сборка

> Electron-shell декомиссирован. Все упоминания `electron/main.js`,
> `electron-builder`, `.next-desktop/standalone`, `lighchat-media://`
> устарели и удалены из репозитория.

1. **Сборка**: `cd mobile/app && flutter build {macos,windows,linux} --release`. Артефакты в `mobile/app/build/`. На macOS дополнительно DMG через `create-dmg` (см. `.github/workflows/desktop.yml`), Windows — MSIX (`flutter pub run msix:create`), Linux — AppImage через `appimagetool`.
2. **Стартовый flow**: `mobile/app/lib/main.dart` инициализирует Firebase, `DesktopShell.initWindow()` (восстановление размера/позиции окна через `window_manager` + `shared_preferences`), затем `runApp`. После uid появляется — `DesktopShell.markDesktopDevice()` пишет `arrayUnion('desktop')` в `users/{uid}.devicePlatforms`, чтобы серверный mirror push (Cloud Function `mirrorPushToFirestore`) активировался.
3. **OAuth-popups**: `firebase_auth_macos` не реализует `signInWithProvider` — на macOS Google/Apple/Yandex/Telegram кнопки показывают понятный SnackBar через `oauthBlockedOnMacOSCheck()` в `auth_screen.dart`. Email + QR-login работают через legacy keychain (см. `flutter_secure_storage` + `MacOsOptions(useDataProtectionKeyChain: false)`) на free Apple ID; для production-сборки нужен paid Apple Developer Program.
4. **Firestore offline persistence**: `cloud_firestore` SDK имеет встроенный `Persistence.localCache()` на всех платформах. Кэш Firestore + flutter_cache_manager (для медиа-картинок чата) дают офлайн-просмотр истории.
5. **Медиа**: `cached_network_image` + `flutter_cache_manager` хранят медиа в userData (`getApplicationCacheDirectory()`); `gallery_save_facade.dart` пишет «сохранение в галерею» через `path_provider.getDownloadsDirectory()` на desktop вместо `gal` (тот mobile-only).
6. **Drag&drop**: `DesktopDropTarget` в `mobile/app/lib/features/desktop_shell/desktop_drop_target.dart` перехватывает файлы, сброшенные на окно, и роутит через `/share` flow.
7. **Push на Windows/Linux**: нет нативного `firebase_messaging` SDK — клиент использует `PushFallbackService` (слушает Firestore коллекцию `users/{uid}/incomingNotifications`, заполняемую серверной `mirrorPushToFirestore` в `onMessageCreated` / `onCallCreated`).
8. **Screenshot protection** для секретных чатов: macOS — `NSWindow.sharingType = .none` через MethodChannel в `MainFlutterWindow.swift`; Windows — `SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)` в `flutter_window.cpp`; Linux — нет API.
