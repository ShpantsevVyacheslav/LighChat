# 04: Runtime Flows

Краткая карта основных пользовательских и системных потоков.

## 1) Auth и профиль

1. Пользователь входит через Firebase Auth (`src/hooks/use-auth.tsx`).
2. Регистрация с фото: исходный кадр после выбора файла сохраняется в Storage как полноразмерный JPEG (до 2048 px по длинной стороне, см. [`uploadUserAvatarPair`](../src/lib/upload-user-avatar-pair.ts)) → `users.avatar`. [`RegisterAvatarCropOverlay`](../src/components/auth/register-avatar-crop-overlay.tsx) дополнительно экспортирует круг ([`getCircularCroppedImageBlob`](../src/lib/circular-crop-to-blob.ts), 512×512) → `users.avatarThumb`. В списках и миниатюрах UI берёт URL из [`userAvatarListUrl`](../src/lib/user-avatar-display.ts) (`avatarThumb` с fallback на `avatar`). Рендер **внутри** `DialogContent`, плюс `showCloseButton={!cropOpen}` и `onOpenChange`, чтобы не закрыть анкету при открытой обрезке.
3. Страница «Мой профиль» ([`dashboard/profile/page.tsx`](../src/app/dashboard/profile/page.tsx), форма [`UserForm`](../src/components/admin/user-form.tsx) с `isProfilePage`): телефон из Firestore нормализуется через [`phoneFormValueFromStored`](../src/lib/phone-utils.ts) и показывается в [`PhoneInput`](../src/components/ui/phone-input.tsx) с той же маской, что при регистрации; аватар — круг + [`RegisterAvatarCropOverlay`](../src/components/auth/register-avatar-crop-overlay.tsx) с `variant="compact"` и [`scopeWithinElement`](../src/contexts/dashboard-main-column-scope.tsx) (размытие и модалка только над правой колонкой [`DashboardMainAndChatRail`](../src/components/dashboard/DashboardMainAndChatRail.tsx), список чатов не затемняется). Контент без обёртки `Card`. Под полями и под аватаром — `FormDescription` / короткий текст как в регистрации.
4. Дубликаты email / телефона / логина: [`register` в use-auth](../src/hooks/use-auth.tsx) возвращает `RegisterResult` с `conflictField`; [`page.tsx`](../src/app/page.tsx) выставляет `react-hook-form` `setError` на поле и красную рамку [`AUTH_GLASS_INPUT_ERROR_CLASS`](../src/components/auth/auth-glass-classes.ts). Ошибка проверки `registrationIndex` без поля — по-прежнему общий баннер.
5. Google: после первого входа профиль считается неполным, пока не заполнены те же поля, что и при email-регистрации (имя ≥2 символов после trim, логин, телефон 11 цифр, email). Критерии — [`isRegistrationProfileComplete`](../src/lib/registration-profile-complete.ts). Форма без пароля — [`completeGoogleProfile`](../src/hooks/use-auth.tsx) и UI-блок [`RegisterDialogFormBlock`](../src/components/auth/register-dialog-form-block.tsx); редирект в [`dashboard/layout`](../src/app/dashboard/layout.tsx) только при полном профиле. Проверки `registrationIndex` с `{ exceptUid }`, чтобы не конфликтовать с собственной записью пользователя. После `setDoc`/`updateDoc` профиля `register` и `completeGoogleProfile` делают `getDoc` и [`setAppUser` в use-auth](../src/hooks/use-auth.tsx), плюс синхронный `appUserRef`. Слушатель `onSnapshot(users/{uid})` **игнорирует снимки из локального кэша** (`metadata.fromCache`), если они откатывают уже полный профиль к неполному — иначе после сохранения анкеты кэш присылал старый документ и снова открывал форму. Если кэш отдаёт **неполный** профиль при входе через Google, перед применением выполняется `getDocFromServer`: при полном документе на сервере подставляется он, чтобы не мигала анкета дозаполнения. Режим модалки (Google без пароля / email с паролем) задаётся флагом `googleProfileCompletionFlow` в контексте. Сразу после `signInWithPopup` / `getRedirectResult` [`finalizeGoogleCredential`](../src/hooks/use-auth.tsx) оборачивает первый `getDoc`/`setDoc` на `users/{uid}` в повторные попытки: `reload` + `getIdToken(true)`, чередование `getDocFromServer` / `getDoc` и пауза при `permission-denied` — стабилизует связку Auth ↔ persistent local cache Firestore.
6. Профиль читается из `users/{uid}`; при отсутствии формируется fallback.
7. Проверяются ограничения (`deletedAt`, `accountBlock`).
8. Обновляются `online/lastSeen` и фоновые presence-сценарии.
9. Смена email в профиле ([`updateUser`](../src/hooks/use-auth.tsx)): сначала вызывается `updateEmail` (Firebase Auth). Если в проекте включена политика «сначала подтвердить новый адрес», API возвращает `auth/operation-not-allowed` с текстом про verify — тогда клиент вызывает `verifyBeforeUpdateEmail` с `continueUrl` на `/dashboard/profile`; в Firestore новый email **не** пишется до подтверждения. После перехода по ссылке из письма email в Auth обновляется; [`use-auth`](../src/hooks/use-auth.tsx) при расхождении Auth vs `users/{uid}.email` делает неблокирующий `updateDoc`, чтобы профиль и [`registrationIndex`](../src/lib/registrationIndexKeys.ts) соответствовали Auth.
10. Mobile Flutter (`mobile/app`): после Google sign-in и в callback `firebaseauth/link` маршрут выбирается через tri-state gate [`getFirestoreRegistrationProfileStatusWithDeadline`](../mobile/app/lib/features/auth/registration_profile_gate.dart) + [`googleRouteFromProfileStatus`](../mobile/app/lib/features/auth/registration_profile_gate.dart): `complete` (`name`, `username`, `email`, `phone`) -> `/chats`, `incomplete` -> `/auth/google-complete`, `unknown` (timeout/транзиентные ошибки) не форсирует редирект на анкету. Логика применяется в [`auth_screen.dart`](../mobile/app/lib/features/auth/ui/auth_screen.dart), [`app_router.dart`](../mobile/app/lib/app_router.dart) и gate списка чатов [`chat_list_screen.dart`](../mobile/app/lib/features/chat/ui/chat_list_screen.dart). Поле телефона в mobile auth-формах вводится с маской `+7 (___) ___-__-__` и нормализуется при сохранении в `+7XXXXXXXXXX`.
11. Mobile чат-экран списка: после успешной авторизации рендер всегда детерминированный — либо список бесед, либо явный empty-state «Пока нет чатов» с CTA `Новая`; аватар в нижней навигации открывает аккаунт-меню ([`chat_account_menu_sheet.dart`](../mobile/app/lib/features/chat/ui/chat_account_menu_sheet.dart)) с рабочими действиями `Профиль` -> `/profile` и `Выйти` -> `/auth`.
12. Mobile настройки чатов: из аккаунт-меню пользователь открывает `/settings/chats` ([`chat_settings_screen.dart`](../mobile/app/lib/features/chat/ui/chat_settings_screen.dart)); экран читает/пишет `users.chatSettings` и `users.customBackgrounds` через [`chat_settings_repository.dart`](../mobile/app/lib/features/chat/data/chat_settings_repository.dart), а пользовательские фоны загружаются в Storage по пути `wallpapers/{uid}/...`.
13. Mobile список чатов: вкладка `Избранное` в [`chat_folder_bar.dart`](../mobile/app/lib/features/chat/ui/chat_folder_bar.dart) работает как shortcut (не фильтр) — открывает личный saved-messages чат пользователя, при отсутствии создаёт его через [`ensureSavedMessagesChat`](../mobile/packages/lighchat_firebase/lib/src/chat_repository.dart) и сразу переходит в `/chats/{id}`. Для saved-chat чтение в правилах ограничено только владельцем (`participantIds.size()==1`).

## 2) Компоновка дашборда и список чатов

1. Во всех разделах дашборда ([`dashboard/layout.tsx`](../src/app/dashboard/layout.tsx) + [`DashboardMainAndChatRail`](../src/components/dashboard/DashboardMainAndChatRail.tsx)): на **md и шире** **слева** колонка [`DashboardChatListColumn`](../src/components/dashboard/DashboardChatListColumn.tsx) (папки, поиск, список диалогов) и под ней [`DashboardBottomNav`](../src/components/dashboard/DashboardBottomNav.tsx) (`variant="chatSidebar"`); **справа** основная область страницы (`children`). На **узком экране** (`max-width: 767px`, см. [`useIsMobile`](../src/hooks/use-mobile.tsx) / брейкпоинт `md`) колонка списка скрыта (`hidden md:flex`), внизу рельса показывается [`DashboardBottomNav`](../src/components/dashboard/DashboardBottomNav.tsx) с `variant="fullWidth"` (иконки разделов + аватар). У рельса **две границы `<Suspense>`**: левая и правая ветки отдельно вызывают [`useDashboardConversationUrl`](../src/hooks/use-dashboard-conversation-url.ts) (`useSearchParams`), чтобы пауза/фолбэк одной колонки не размонтировал список чатов в другой. Левая ветка по-прежнему монтируется на мобильных (невидима), чтобы не менять подписки без отдельной задачи на ленивую загрузку списка.
2. Открытый чат: query [`conversationId`](../src/lib/dashboard-conversation-url.ts) в URL; **правая** колонка рендерит [`DashboardOpenChatView`](../src/components/dashboard/DashboardOpenChatView.tsx) вместо `children`. Маршруты утилит вроде `/dashboard/chat/forward` не используют этот query для колонки с контентом ([`isDashboardChatUtilityPath`](../src/lib/dashboard-conversation-url.ts)).
3. UI берёт `userChats/{uid}.conversationIds`.
4. Документы чатов подписываются по id (`useConversationsByDocumentIds`), без общего list-query.
5. В `ChatWindow` читаются `conversations/{id}/messages` и связанные поддокументы.
6. Изменения обновляют метаданные чата (last message, unread, thread counters).
7. Удалённые сообщения ([`ChatMessageItem`](../src/components/chat/ChatMessageItem.tsx)): при `isDeleted` сетка вложений в расчёте ширины не участвует (`hasGridVisualMedia` = false), плашка «Сообщение удалено» выравнивается `self-end` / `self-start` как исходящие/входящие. Геолокация: [`MessageLocationCard`](../src/components/chat/parts/MessageLocationCard.tsx) вне цветного пузыря; время — [`MessageStatus`](../src/components/chat/parts/MessageStatus.tsx) с `overlay` на превью карты; «живое» превью скрывается при истечении [`liveSession.expiresAt`](../src/lib/types.ts) или когда у отправителя нет видимой трансляции в `users/{uid}.liveLocationShare` — см. [`isChatLiveLocationMessageStillStreaming`](../src/lib/live-location-utils.ts) (таймер `setTimeout` перерисовки после дедлайна; ручной «Остановить» обновляет профиль, текст пузыря совпадает с истечением таймера).
8. Профиль чата / контакта ([`ChatParticipantProfile`](../src/components/chat/ChatParticipantProfile.tsx)): одна правая шторка (`Sheet`, [`WA_CONVERSATION_UTILITY_SHEET_CONTENT_CLASS`](../src/components/chat/profile/ParticipantProfileWhatsAppLayout.tsx)) с `bg-background` / семантическими цветами текста, чтобы в теме «Авто» фон и контраст совпадали с палитрой от глобальных обоев (`AppThemeController`). Пункты «Медиа / Избранное / …» подменяют содержимое шторки (`profileSubMenu` + `conversation-pages/*Panel.tsx`); список действий — компактные [`WaMenuRow`](../src/components/chat/profile/ParticipantProfileWhatsAppLayout.tsx) / [`WaMenuSection`](../src/components/chat/profile/ParticipantProfileWhatsAppLayout.tsx) без групповых «карточек» и без горизонтальных разделителей между строками. Вкладка медиа ([`ConversationMediaPanel`](../src/components/chat/conversation-pages/ConversationMediaPanel.tsx)) открывает тот же полноэкранный [`MediaViewer`](../src/components/chat/media-viewer.tsx), что и лента чата; сетка видеокружков — [`VideoCirclePlayer`](../src/components/chat/VideoCirclePlayer.tsx) с `layout="grid"`, чтобы превью не наезжали друг на друга. Скролл меню профиля — нативный `overflow-y-auto` с `flex-1 min-h-0 basis-0` и `touch-pan-y` на области прокрутки (не на корне шторки): иначе на iOS вертикальный жест не доходит до списка. Жест «закрыть свайпом вправо» — только на верхнюю полосу. Прямые URL открывают ту же панель через [`ConversationUtilityRouteSheet`](../src/components/chat/conversation-pages/ConversationUtilityRouteSheet.tsx) и [`buildDashboardChatOpenUrl`](../src/lib/dashboard-conversation-url.ts).
9. Тема «Авто» (next-themes `chat`, [`AppThemeController`](../src/components/AppThemeController.tsx)): CSS-переменные акцента и класс `.dark` на `html` считаются по **глобальному** фону `users/{uid}.chatSettings.chatWallpaper`, который отдаёт [`useSettings`](../src/hooks/use-settings.ts) (мерж с дефолтами). В хуке значение нормализуется: пустая строка и прочие «пустые» значения → `null`, чтобы палитра и контраст не залипали на устаревшем состоянии. Обои **конкретного** чата из `chatConversationPrefs` влияют на [`ChatWallpaperLayer`](../src/components/chat/ChatWallpaperLayer.tsx) в окне переписки, но не подменяют источник для режима «Авто» на уровне приложения. Для **http(s)** URL картинки выборка цвета в canvas идёт через same-origin [`/api/wallpaper-for-theme`](../src/app/api/wallpaper-for-theme/route.ts) ([`wallpaperImageUrlForThemeSampling`](../src/lib/wallpaper-theme-image-url.ts) + [`sampleWallpaperImageAverageRgb`](../src/lib/chat-app-theme.ts)): прямой `Image` с Storage в Safari/WebKit часто блокируется CORS при чтении пикселей, прокси обходит это без настройки CORS на бакете.

## 3) Отправка сообщений

1. Клиент пишет сообщение в `conversations/{id}/messages/{messageId}`.
2. Обновляет parent `conversations/{id}` (последнее сообщение/счётчики/реакции).
3. Firestore trigger `onmessagecreated` отправляет FCM другим участникам.
4. Для тредов trigger `onthreadmessagecreated` шлёт отдельные уведомления.
5. **Черновик сообщения (web):** неотправленный HTML ввода и контекст «ответ» сохраняются в `localStorage` per user (`src/lib/chat-message-draft-storage.ts`) при уходе с чата (`pagehide`, размонтирование поля ввода, смена `conversationId` / ключа треда). Ключ основного чата — `conversation.id`; для треда — `t:{conversationId}:{parentMessageId}`. После успешной отправки или правки сообщения запись очищается. В списке бесед [`ConversationItem`](../src/components/chat/ConversationItem.tsx) показывает метку «Черновик» и превью ([`useChatMainDraftPreview`](../src/hooks/use-chat-main-draft-preview.ts)). Вложения файлов в черновик не сериализуются.

**Персональные настройки push:** при сборке payload учитываются `users/{uid}.notificationSettings` (`muteAll`, тихие часы с IANA-зоной `quietHoursTimeZone`, глобальный `showPreview`, `soundEnabled` → флаг `silent` в `data`) и `users/{uid}/chatConversationPrefs/{conversationId}` (`notificationsMuted`, переопределение превью). Логика в [`functions/src/lib/push-notification-policy.ts`](../functions/src/lib/push-notification-policy.ts); доставка батчами — [`fcm-send-data-batches.ts`](../functions/src/lib/fcm-send-data-batches.ts). Аналогичные правила для уведомлений «добавили в группу» (`onconversationcreated` / `onconversationupdated`). Входящий звонок отменяется только при `muteAll`. Клиент: [`public/firebase-messaging-sw.js`](../public/firebase-messaging-sw.js) читает `data.silent`; foreground — [`use-notifications.ts`](../src/hooks/use-notifications.ts) подавляет toast при тех же условиях (текст уже приведён сервером).

### Сквозное шифрование (E2E), опционально

- Протокол v1: P-256 ECDH + AES-256-GCM; описание в [`src/lib/e2ee/protocol.ts`](../src/lib/e2ee/protocol.ts). Ключ устройства в IndexedDB; публичный ключ — `users/{uid}/e2ee/device`.
- Включение в чате: [`enableE2eeOnConversation`](../src/lib/e2ee/enable-conversation.ts) создаёт `conversations/{id}/e2eeSessions/{epoch}` с обёртками ключа для каждого участника и выставляет флаги на `Conversation`. Для **текущего пользователя** в обёртку берётся локальный SPKI из IndexedDB (тот же, что у приватного ключа), а не повторное чтение из Firestore — иначе сразу после `publish` клиентский кэш мог отдать старый ключ и отправка не могла бы сделать unwrap.
- Сообщения с `e2ee` хранят ciphertext; `text` с секретом не используется. Расшифровка только в клиенте ([`useE2eeConversation`](../src/hooks/use-e2ee-conversation.ts), [`ChatWindow`](../src/components/chat/ChatWindow.tsx), [`ThreadWindow`](../src/components/chat/ThreadWindow.tsx)).
- Гибрид: старые сообщения без `e2ee` остаются plaintext; новые после включения — только ciphertext.
- Авто-E2E при создании личного чата: [`tryAutoEnableE2eeNewDirectChat`](../src/lib/e2ee/enable-conversation.ts) если включено у пользователя и/или в `platformSettings/main.e2eeDefaultForNewDirectChats`.
- Ручное включение/выключение для **конкретного личного** чата: шторка профиля [`ChatParticipantProfile`](../src/components/chat/ChatParticipantProfile.tsx) → пункт «Шифрование» → [`ConversationEncryptionPanel`](../src/components/chat/conversation-pages/ConversationEncryptionPanel.tsx); снятие флагов — [`disableE2eeOnConversation`](../src/lib/e2ee/disable-conversation-e2ee.ts). История с полем `e2ee` по-прежнему расшифровывается по эпохе в сообщении, даже если у чата `e2eeEnabled: false`.
- Пересылка сообщений с `e2ee` отключена ([`forward/page.tsx`](../src/app/dashboard/chat/forward/page.tsx)). Геолокация и опросы в E2E-чате блокируются на клиенте до отдельной доработки.

## 4) 1:1 звонки

1. Создаётся `calls/{callId}` с caller/receiver и offer.
2. Trigger `oncallcreated` обновляет `userCalls` и отправляет входящий push.
3. Клиенты обмениваются offer/answer/candidates через `calls/{id}` и `calls/{id}/candidates`.
4. Завершение звонка обновляет статус и timestamps в документе call.

## 5) Встречи (meetings)

1. Создаётся `meetings/{meetingId}`.
2. Перед входом открывается `JoinMeeting`; на публичной ссылке `/meetings/[meetingId]` поле имени показывается и для анонимных гостей (Firebase anonymous), чтобы имя задавалось до подключения WebRTC.
3. Участники входят в `participants`; trigger обновляет `userMeetings`.
4. ICE-настройки для meetings и 1:1 calls берутся из общего `src/lib/webrtc-ice-servers.ts`: клиент запрашивает `/api/webrtc/ice`, где сервер получает временные TURN-credentials (Metered) по приватным ENV `METERED_DOMAIN`/`METERED_API_KEY`; при ошибке используется STUN fallback.
5. Signaling: `meetings/{id}/signals`.
6. Встроенный чат: `meetings/{id}/messages`; опросы: `meetings/{id}/polls`.
7. Для приватных встреч запросы в `meetings/{id}/requests` через callable функции.

## 6) Админ и служебные потоки

- Callable admin endpoints (`createNewUser`, `updateUserAdmin`, backfill-операции).
- Периодический scheduler `checkUserPresence` чистит stale presence/meeting records.
- Server actions в `src/actions/*` выполняют read-heavy/privileged операции для UI-панелей.

## 7) iOS PWA performance guards

- Оболочка мобильного чата (`dashboard/layout.tsx`) задаёт высоту через `--lc-visual-vh` из [`useVisualViewportCssVars`](../src/hooks/use-visual-viewport-css-vars.ts): при клавиатуре берётся `visualViewport.height` + `offsetTop`, иначе `max(vv.height, window.innerHeight)` — после возврата PWA из фона WebKit иногда отстаёт по `visualViewport`, из‑за чего без `innerHeight` нижняя навигация оказывалась не у края экрана; дополнительно `visibilitychange` / `pageshow` и отложенные пересчёты синхронизируют высоту.
- Бейдж непрочитанных сообщений рассчитывается только в chat-маршрутах, чтобы не держать тяжёлые realtime-подписки на каждом экране dashboard.
- PWA onboarding выполняет длинные FCM-подписки в фоне и не блокирует UI длительным ожиданием.
- Overlay звонков подключается с небольшой отложенной инициализацией после входа в dashboard, чтобы разгрузить cold-start.
- В meetings ограничен объём realtime-данных на старте (лимиты истории чата и опросов).
- В meetings listeners для опросов/зала ожидания активируются по соответствующей вкладке sidebar, чтобы не держать лишние realtime-подписки постоянно.
- В chat list typing-listener на `conversations/{id}/typing` включается всегда для выбранного чата и только для видимых карточек при активной вкладке для остальных, чтобы вернуть индикатор "Печатает..." без постоянной нагрузки.
- В fullscreen media viewer чата на iOS touch-жесты разделены по зонам: media-зона отдаёт pinch/pan в `react-zoom-pan-pinch`, а swipe-down закрытия обрабатывается только из overlay-зоны.
- В ChatWindow/ThreadWindow догрузка истории вверх фиксирует `scrollTop/scrollHeight` перед увеличением window-limit и компенсирует смещение после рендера, чтобы избежать прыжков верстки при prepend старых сообщений.
- В ChatWindow/ThreadWindow, помимо `startReached`, включён fallback-триггер по `scrollTop <= 40px` с cooldown, чтобы догрузка истории стабильно срабатывала на устройствах, где виртуализатор не всегда эмитит верхнее событие.

## 8) Desktop (Electron): локальный UI и офлайн‑просмотр

1. **Собранное приложение** не грузит интерфейс с Firebase Hosting: в `electron/main.js` поднимается вложенный процесс Next **production** из **`.next-desktop/standalone/server.js`**. Сборка для десктопа идёт с `LIGHCHAT_DESKTOP_BUILD=1` (отдельный `distDir: '.next-desktop'` и `output: 'standalone'`), чтобы не делить `.next` с `next dev` и не ловить гонки/битые манифесты при трассировке; вшитый в `server.js` `distDir` указывает на **`./.next-desktop`** относительно каталога standalone, поэтому после `next build` скрипт `build:clean` копирует `.next-desktop/static` → **`standalone/.next-desktop/static`** (не в `.next/static`) и `public` → `standalone/public`.
2. Процесс стартует с `cwd` = каталог standalone и `ELECTRON_RUN_AS_NODE=1`, чтобы слой Node внутри сборки Electron выполнял `server.js` как обычный Node. Локальный URL — `http://localhost:3434` (вместо IP), чтобы OAuth/Firebase не упирался в отдельную авторизацию `127.0.0.1`. Для резолва модулей рантайм дополнительно прокидывает `NODE_PATH` с fallback-кандидатами (`standalone/node_modules`, `app.asar/node_modules`, `app.asar.unpacked/node_modules`) — это защищает запуск, если при переносе приложения отсутствует `standalone/node_modules`.
3. Окна `window.open` в Electron по умолчанию блокируются и уходят во внешний браузер, но для Google OAuth оставлен allowlist (`accounts.google.com`, `project-72b24.firebaseapp.com/__/auth/*`), чтобы `signInWithPopup` в desktop мог завершаться внутри приложения.
4. **Данные Firestore** при доступном IndexedDB уже кэшируются клиентским SDK (`persistentLocalCache` в [`src/firebase/index.ts`](../src/firebase/index.ts)); без сети доступно то, что было загружено ранее.
5. **Медиа из Storage** для Electron дополнительно кэшируются на диск в userData через IPC [`electron/media-cache.js`](../electron/media-cache.js), а в рендерер отдаются через кастомный протокол `lighchat-media://` (регистрация в [`electron/main.js`](../electron/main.js)). Это убирает ограничения загрузки `file://` из `http://localhost:*` и снижает CORS-проблемы для HEIC/HEIF цепочки [`useChatAttachmentDisplaySrc`](../src/components/chat/use-chat-attachment-display-src.ts) / [`src/lib/heic-heif-convert.ts`](../src/lib/heic-heif-convert.ts).
6. Упаковка: `electron-builder` включает `.next-desktop/standalone` и распаковывает его из asar (`asarUnpack`), чтобы нативные зависимости внутри standalone не ломались. Путь к серверу в рантайме резолвится относительно **`app.asar.unpacked`** (а не `app.asar`), иначе `server.js` не находится и окно остаётся пустым.
