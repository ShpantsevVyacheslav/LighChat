## TL;DR

- **5 Critical, 12 High, 14 Medium, 9 Low/Info** (37 находок).
- **Топ-5 рисков:**
  1. **`GoogleService-Info.plist` с `API_KEY` коммитится в git, без HTTP-referrer
     restrictions** — публичный доступ к проекту `project-72b24` для атакующего
     (CR-001).
  2. **App Check полностью отсутствует** ни в одной callable Cloud Function,
     ни в SDK web/mobile — любой может звать `requestQrLogin`,
     `signInWithTelegram`, читать Firestore с публичным API key (CR-002).
  3. **Дашборд подписывается на ВСЮ коллекцию `users` через `onSnapshot`** —
     при 10k DAU это ≈ 10 000 reads на каждый вход + realtime — десятки $/мес
     даром, плюс утечка персональных данных всем пользователям (CR-003).
  4. **`deleteAccount` не удаляет файлы в Storage** (`avatars/`, `wallpapers/`,
     `chat-attachments/{convId}/`, `users/{uid}/sticker-packs/...`) — нарушение
     GDPR + бесконечно растущий счёт за хранение (CR-004).
  5. **Удаление `conversations/{id}` не каскадит на subcollections и Storage** —
     осиротевшие `messages/*`, `e2eeSessions/*`, `chat-attachments/{id}/*`
     остаются навсегда (CR-005).
- **Оценка экономии при устранении cost-рисков:** на сценарии 1 000 DAU ×
  100 сообщений/чат/день ≈ **$80–180 / мес** (см. секцию по бюджету); рост
  $/мес квадратичный по числу пользователей из-за `collection('users')`
  на дашборде, поэтому при 10k DAU экономия достигает $400–1000 / мес.
- **CI/CD сломан**: единственный workflow — `dart.yml`, который пытается
  `dart pub get` на корне репозитория (где нет `pubspec.yaml`). Никаких
  typecheck / lint / vitest / firestore-rules unit-тестов на PR в main нет.

## Critical

### [CR-001] `GoogleService-Info.plist` с production API key закоммичен в git

  `mobile/app/ios/Runner/GoogleService-Info.plist:9`,
  `mobile/app/macos/Runner/GoogleService-Info.plist`.
- **Что:** `.gitignore:73` содержит `**/GoogleService-Info.plist`, но три копии
  всё равно tracked. Файлы содержат `API_KEY = AIzaSyBfedXYICIrF0VPOXkmqB72QoW6J3fEXpg`,
  `CLIENT_ID`, `STORAGE_BUCKET`. По
  `docs/security-audit-2026-05-followups.md` Browser API key до сих пор без
  HTTP-referrer / iOS bundle / Android SHA-1 ограничений (раздел «Ограничить

- **Грозит:** атакующий, склонировавший репо или прошедший через GitHub
  search по `AIza`, получает полностью валидный ключ для Firebase web/iOS, может
  напрямую звать Identity Toolkit (`signUp`, `signInWithPassword`), Firestore,
  Storage REST с любого хоста. В сочетании с CR-002 (нет App Check) —
  бесконечный скам/ботнет на стоимость владельца.
- **Фикс:** (1) `git rm --cached GoogleService-Info.plist
  mobile/app/ios/Runner/GoogleService-Info.plist
  mobile/app/macos/Runner/GoogleService-Info.plist`, ротировать API key в Cloud
  Console (создать новый, удалить старый), (2) выставить **Application
  restrictions** на каждом ключе (HTTP referrers для web ключа, iOS bundle id,
  Android SHA-1) и **API restrictions** только на нужные API, (3) добавить
  `git filter-repo` чистку истории либо принять «ключ скомпрометирован,
  отозван». **(effort: M)**

### [CR-002] App Check не подключён ни в одной callable / SDK
- **Файл:** ни одного `enforceAppCheck: true` в `functions/src/triggers/http/*`
  (`grep -r enforceAppCheck functions/src` пусто); ни одного
  `initializeAppCheck` в `src/firebase/index.ts`, `mobile/app/lib/`. Текущие
  pre-auth callable: `requestQrLogin`, `signInWithTelegram`, `confirmQrLogin`
  имеют только per-IP rate limit `10/min`
  (`functions/src/triggers/http/requestQrLogin.ts:158`,
  `functions/src/triggers/http/signInWithTelegram.ts:163`).
- **Грозит:** (a) DoS по pre-auth callable — каждый вызов пишет документ в
  `qrLoginSessions`/`rateLimits`, IPv6 ротация атакующего тривиально обходит
  rate limit; (b) злоупотребление quota Firestore/Storage с фиксированного
  webApiKey без референта; (c) bot-регистрации (Telegram widget), создание
  фейковых `tg_*` UID; (d) полностью открытая запись в `e2eeDevices/{deviceId}`
  и `users/{uid}/devices/{deviceId}` через REST для скомпрометированного
  device.
- **Фикс:** реальный App Check rollout (web reCAPTCHA Enterprise, iOS
  DeviceCheck / AppAttest, Android Play Integrity). Сначала monitor mode 1
  неделя → enforce. В callable добавить `{ enforceAppCheck: true }`. **(effort: L)**

### [CR-003] Dashboard грузит всю коллекцию `users` через `onSnapshot`
- **Файл:** `src/components/dashboard/DashboardChatListColumn.tsx:88-93`
  (`collection(firestore, 'users')`), плюс
  `src/components/dashboard/DashboardOpenChatView.tsx:97`,
  `src/app/dashboard/chat/[conversationId]/threads/page.tsx:28`,
  `src/app/games/durak/[gameId]/page.tsx:20`,
  `src/components/profile/BlockedUsersPageClient.tsx:31-35` — все вешают
  `useCollection<User>` без `where`/`limit`.
- **Что:** `firestore.rules:379-384` разрешает любому signed-in читать любой
  `users/{userId}` (если не заблокирован), поэтому Firestore возвращает
  ровно `count(users)` документов. На дашборд это листенер, обновляемый при
  любом изменении профиля любого пользователя.
- **Грозит (бюджет):** при 10 k активных пользователей — `10 000 reads × DAU ×
  hits/день` плюс realtime-апдейты. Минимум **30–80 М reads/мес** на одной этой
  подписке, $0.06/100k reads → **$18–48 / мес уже сейчас**, и квадратично растёт
  с базой. Также утечка PII (телефоны, email, аватары всех) — формально доступ
  по правилам разрешён, но фактически клиент кэширует это и отдаёт в любой
  баг-репорт / Crashlytics.
- **Фикс:** заменить `collection('users')` на адресные подписки по конкретным
  uid (как уже сделано в `useUsersByDocumentIds` для `[participantIds, ...]`).
  Контакты/группы — `useUsersByDocumentIds(contactIdsForSearch ∪ участники
  чатов)`. Убрать «глобальный список» из дашборда. **(effort: M)**

### [CR-004] `deleteAccount` не удаляет ничего в Cloud Storage и не отзывает FCM
- **Файл:** `functions/src/triggers/http/deleteAccount.ts:202-268`. Удаляются
  только `users/`, `userChats/`, `userSecretChats/`, `userContacts/`,
  `userCalls/`, `userMeetings/` в Firestore + Auth user. Поиск по файлу:
  `grep -E "fcmTokens|getStorage|bucket\(" functions/src/triggers/http/deleteAccount.ts`
  пусто.
- **Что не удаляется:** `avatars/{uid}/...`, `wallpapers/{uid}/...`,
  `users/{uid}/sticker-packs/...`, `users/{uid}/backgrounds/...`,
  `chat-attachments/{convId}/...` (там, где `convId` — DM с уходящим), ICE/CALL
  audio. FCM токены не удаляются (поле `users/{uid}.fcmTokens` рекурсивно
  удалится вместе с doc, но сами регистрации в FCM проекте — остаются и могут
  получать старые pushes если token используется на чьём-то устройстве).
- **Грозит:** (a) GDPR/«право на удаление» нарушение — медиа, аватары,
  пользовательские стикеры остаются в Storage навсегда; (b) бесконечный рост
  Storage-счета; (c) отозванный пользователь продолжает получать push на
  старое устройство, если Auth uid пересоздаётся.
- **Фикс:** в `deleteAccount` (1) `bucket.deleteFiles({ prefix: 'avatars/${uid}/'
  })`, аналогично `wallpapers/`, `users/${uid}/sticker-packs/`,
  `users/${uid}/backgrounds/`; (2) для DM (`deletedDmConversationsCount`)
  одновременно `bucket.deleteFiles({ prefix: 'chat-attachments/${convId}/' })`;
  (3) `messaging().unsubscribeFromTopic(token, ...)` или удалять
  токены через FCM IID API; (4) перенести длинную работу в Pub/Sub task,
  чтобы callable не выходила за 60s. **(effort: M)**

### [CR-005] Удаление `conversations/{id}` не каскадит на subcollections и Storage
- **Файл:** `functions/src/triggers/firestore/onConversationDeleted.ts:14-69`.
  Триггер чистит только `userChats/{uid}.conversationIds`, `secretChats/`,
  `userSecretChats/`. **Не вызывает `recursiveDelete` на
  `conversations/{id}`**, поэтому `messages/*`, `polls/*`, `e2eeSessions/*`,
  `secretAccess/*`, `secretMediaViewState/*`, `gameLobbies/*`, `tournaments/*`
  остаются. Storage файлы по `chat-attachments/{convId}/` тоже остаются.
- **Что:** `firestore.rules:731-735` разрешает удалить parent doc любому
  участнику DM (или admin/creator группы). После delete пользователь больше не
  «participant» (его в parent docs больше нет), но subcollection remains
  читаемой только тем, у кого `members/{uid}` ещё лежит — а у удалённого
  conversation `members/*` чистятся
  (`onConversationDeleted -> deleteAllMemberDocsForConversation`). Однако
  данные **физически остаются и тарифицируются**.
- **Грозит:** (a) бесконечный рост стоимости Firestore/Storage (раз и
  навсегда — никогда не очищается); (b) если позже добавится коллизия id
  (например, новый `dm_aaa:bbb`), новые сообщения смешаются со старыми; (c)
  E2EE: лежат `e2eeSessions/*` со старыми ciphertext-ключами.
- **Фикс:** в `onConversationDeleted` добавить
  `await db.recursiveDelete(db.doc('conversations/${id}'))` (или удалить
  каждый subcol-набор с pagination 500), затем
  `await getStorage().bucket().deleteFiles({ prefix: 'chat-attachments/${id}/'
  })` и `chat-attachments-enc/${id}/`. Для group-чатов (где остальные
  участники должны увидеть «чат удалён») — оставить шаг как async cleanup, но
  обязательно его выполнить. **(effort: M)**

## High

### [H-001] `meeting-attachments/*` читает любой signed-in (cross-tenant)
- **Файл:** `storage.rules:158-168`. `allow read: if signedIn();` на
  `meeting-attachments/{meetingId}/...`. Гость A в meeting M1 знает полный
  путь к файлу из meeting M2 — может качать.
- **Грозит:** утечка медиа-вложений между разными митингами; гость, попавший в
  один митинг, получает доступ к асетам всех остальных при наличии полного
  пути (paths берутся из Firestore, который для участников открыт).
- **Фикс:** `allow read: if isMeetingMemberStorage();` (как уже сделано для
  write). Если нужен fallback для фоновых ассетов — отдельный path
  `meeting-assets/`. **(effort: S)**

### [H-002] Анонимные гости видят все митинги и все профили
- **Файл:** `firestore.rules:511-512` (`match /meetings/{meetingId} { allow read:
  if isSignedIn(); }`) и `firestore.rules:379-384`. Anonymous Auth (раздел
  `isAnonymousGuest`) проходит проверку `isSignedIn()`.
- **Грозит:** гость, открывший произвольный `/meetings/{id}`, может
  перебирать `meetingId` и читать `hostId`, `name`, список приглашённых
  (если в схеме есть). Аналогично — `users/{uid}` у любого, кто гостя не
  заблокировал. Скан персональных данных после `signInAnonymously()`.
- **Фикс:** для `meetings/{id}` — отдельная ветка чтения для anonymous: только
  если `meetingId` совпадает с `request.path` deep-link сессии. Для `users/`
  — добавить `request.auth.token.firebase.sign_in_provider != 'anonymous'`.
  **(effort: M)**

### [H-003] `qrLoginSessions/{sessionId}` публично читается без auth
- **Файл:** `firestore.rules:327-331` (`allow read: if true;`).
- **Что:** реализовано осознанно (см. комментарий в правилах) под предлогом
  «sessionId — 192-битный секрет». Зашифрованный `customTokenCipher`
  защищает компромисс ключа, но **`ip`, `userAgent`, `country`, `city`,
  `deviceLabel`, `ephemeralPubKeySpki` лежат в plaintext** — атакующий, узнав
  sessionId (например, из логов прокси/корпоративного TLS-MitM/share-link),
  получает PII запросившего устройства.
- **Грозит:** деанон по «привязке нового устройства»: IP+city+UA владельца,
  ещё до approval.
- **Фикс:** хранить IP/UA не в самом документе `qrLoginSessions`, а в
  `users/{uid}/devices/{deviceId}` (private), туда писать после approve как
  делает `confirmQrLogin.ts:189-211`. До approve хранить только хэшированный
  fingerprint. **(effort: S)**

### [H-004] `registrationIndex` без auth — энумерация phone/email/login → uid
- **Файл:** `firestore.rules:337-360`.
  `allow read: if request.auth == null || resource == null || (...)` — при
  `auth == null` отдаются все документы. Имя документа — нормализованный ключ
  `tel:+7…` / `email:foo@bar` / `login:...` (см. `registrationIndexKeys.ts`).
- **Грозит:** атакующий, имея список телефонов («известные клиенты компании»),
  через `getDoc('registrationIndex/tel:+7...')` без auth получает соответствие
  phone → LighChat uid. uid → доступ к `users/{uid}` после
  `signInAnonymously` (см. H-002), и таким образом — имя/аватар/город (если в
  публичном профиле).
- **Фикс:** убрать `request.auth == null` ветвь, перенести проверку «свободен ли
  телефон» на серверный callable `checkRegistrationKeyAvailable`. Можно
  отдавать только boolean без uid. **(effort: M)**

### [H-005] Логи pre-auth callable могут содержать PII
- **Файл:** `functions/src/triggers/http/signInWithTelegram.ts:185-196`
  (`process.env.TELEGRAM_DEBUG_LOGIN_INFO === '1'` пишет «redacted» payload —
  но `first_name/last_name/username/photo_url` всё ещё попадают),
  `functions/src/triggers/http/requestQrLogin.ts:185-188` (`stack`

  `src/hooks/use-meeting-webrtc.ts` (mlog.* пишет участников и snapshot keys

- **Грозит:** Cloud Logging тарифицируется + хранит PII; первое — деньги,
  второе — потенциальный GDPR-инцидент при breach логов.
- **Фикс:** убрать `TELEGRAM_DEBUG_LOGIN_INFO`, не логировать первые

### [H-006] N+1 в `onMessageCreated`: per-recipient `getDoc(users)` + `getDoc(prefs)`
- **Файл:** `functions/src/triggers/firestore/onMessageCreated.ts:99-138`.
  Для группового чата на 100 участников = 200 reads на каждое сообщение, плюс
  ещё один read для `users/{senderId}`. Триггер вызывается для **каждого**
  созданного документа, включая system-маркеры E2EE.
- **Грозит:** $$ при активной группе. 100 сообщений в группе × 100 человек ×
  2 reads = 20 000 reads/день/группа. На 100 групп — 2 М reads/день =
  60 М/мес = $36/мес.
- **Фикс:** (1) `db.getAll(...refs)` — батч на recipient docs; (2) пропускать
  push для system-сообщений рано (`senderId === '__system__'`); (3) кэшировать
  prefs внутри одной функции если идентичны. **(effort: M)**

### [H-007] `checkUserPresence` каждую минуту скан коллекций без `limit()`
- **Файл:** `functions/src/triggers/scheduler/checkUserPresence.ts:37-104`.
  Шаги 1, 2, 3, 4 делают `.where(...).get()` без `limit()`.
- **Грозит:** при 100k registered users c `online == true` (после массового
  входа) — 100k reads × 1440 мин/день = 144 М reads/день только на presence
  cleanup. Реально в фоне у проекта в любом случае сотни тысяч reads/день
  «бесплатно».
- **Фикс:** `.limit(500)` на каждый шаг + цикл while; и/или поднять окно до
  «every 5 minutes» (мобильные клиенты heartbeat 20 сек × 4.5 = 90 сек, можно
  спокойно sweep раз в 5 мин). **(effort: S)**

### [H-008] `cleanupGuestAccounts` сканирует ВСЕХ Auth users каждый час
- **Файл:** `functions/src/triggers/scheduler/cleanupGuestAccounts.ts:51-77`.
  Цикл `listUsers(1000, pageToken)` без признака «только гости» — Firebase
  Auth Admin не умеет фильтровать по `providerData empty`. На 1М users: 1000
  страниц × 24 раза/день = 24 000 listUsers calls/day.
- **Грозит:** $0/мес для Firebase Auth Admin (бесплатно), но runtime функции
  ↑ → cold-start backoff/quota; со временем функция начнёт уходить за `60s`
  timeout и не будет дочищать гостей.
- **Фикс:** держать TTL-индекс гостей в Firestore (`guestAccounts/{uid} {
  expireAt }`) — пишет `onUserCreated`, scheduler читает только этот
  индекс. **(effort: M)**

### [H-009] CSP в Report-Only режиме — реально не защищает
- **Файл:** `src/middleware.ts:26` (`const CSP_REPORT_ONLY = true`). При этом
  есть `dangerouslyAllowSVG: true` в `next.config.js:105` и
  `images.contentSecurityPolicy = "script-src 'none'"` — но это только для
  next/image, не для inline SVG в Storage URLs.
- **Грозит:** XSS-регрессия (например, обход DOMPurify на новой версии TipTap)
  не блокируется. SVG из Storage с встроенным `<script>` — защита только
  через storage rules (`isAcceptableImageType` matches `image/*` — пропускает
  `image/svg+xml`? — нет, регулярка не включает svg, поэтому write блокирован,
  но **stickers** пропускают `'image/svg+xml'` через `chat-attachments/`?).
- **Фикс:** перевести CSP в enforcing после 1 недели observation; запретить
  `image/svg+xml` в storage.rules `isAcceptableChatMediaType`. **(effort: S)**

### [H-010] `MessageText` отдаёт raw HTML на server-render
- **Файл:** `src/lib/sanitize-message-html.ts:11`
  (`if (typeof window === 'undefined') return html;`). При SSR клиентских
  компонентов (`'use client'` всё равно рендерится один раз на сервере для
  hydration HTML) функция возвращает `html` без санитизации.
- **Грозит:** если SSR-кэш доберётся до user-supplied HTML (например,
  pre-rendered share-page чата с предпросмотром последнего сообщения), HTML
  попадёт в HTML-ответ браузеру до hydration → XSS до момента, как клиент
  применит DOMPurify.
- **Фикс:** заменить на `isomorphic-dompurify` или вернуть `''` на сервере и
  гидрировать только на клиенте. Сейчас по факту дашборд не SSR-рендерит
  сообщения, но это footgun. **(effort: S)**

### [H-011] Нет idempotency на `onMessageCreated` push
- **Файл:** `functions/src/triggers/firestore/onMessageCreated.ts:140-150`.
  Cloud Functions v2 имеет `at-least-once` доставку — при retry трeggera

- **Грозит:** дубль push-уведомления. На клиенте FCM `tag: conversationId`
  частично спасает, но не на iOS APNs до доставки.
- **Фикс:** в начало триггера `await db.doc('pushDelivered/${messageId}').create(...)`
  внутри транзакции; если уже create-нут — return. **(effort: S)**

### [H-012] `update conversations/{id}` нет защиты `participantIds` от
расширения нечленом
- **Файл:** `firestore.rules:717-725`. Условие `update`:
  `uidIsConversationParticipantFromData(resource.data)` — то есть участник
  ДО апдейта. Однако новое значение `participantIds` (`request.resource.data`)
  не проверяется на «не выкинул ли я остальных» / «не добавил ли я кого-то
  без приглашения».
- **Грозит:** участник группы может одним update'ом вытолкнуть остальных
  (`participantIds = [self]`) или добавить кого угодно. Сейчас от этого

### [M-001] `useConversationsByDocumentIds`: 1 onSnapshot на каждый чат
- **Файл:** `src/firebase/firestore/use-conversations-by-document-ids.tsx:58-90`.
  По одному документ-листенеру на каждый conversationId из `userChats`.

  (тарифицируются как `min(1 doc/listener)`). Поверх этого — реальные апдейты

  reads/day/user × 1 000 DAU = 10 М reads/день = **$180/мес**.
