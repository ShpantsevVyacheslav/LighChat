# Hotfix: fallback-триггер догрузки истории чата у верхней границы

## Контекст

В `ChatWindow` и `ThreadWindow` догрузка старых сообщений завязана на `startReached` (Virtuoso) + увеличение `displayLimit`.

Симптом в текущем регрессе:

- при скролле в верх списка `Загрузка истории...` не появляется;
- старые сообщения не подгружаются;
- значит `handleLoadMore` не срабатывает в ряде сценариев (включая iOS).

## Цель

1. Восстановить надежный запуск догрузки при достижении верха.
2. Не допустить скачков верстки (сохранить текущую компенсацию scroll).
3. Сделать фикс минимально инвазивным без рефакторинга data-модели.

## Принятое решение

Оставляем текущую архитектуру (`displayLimit` + `onSnapshot`) и добавляем резервный trigger по фактическому `scrollTop`:

- основной путь: `startReached` (как сейчас),
- fallback: ручная проверка `scrollTop <= TOP_LOAD_THRESHOLD_PX`.

Добавляется lock/debounce, чтобы один «подход к верху» не запускал несколько параллельных догрузок.

## Почему не полный рефактор сейчас

- Полный переход на cursor-pagination (`getDocs + startAfter`) надежнее стратегически, но крупнее по объёму и рискам для текущей задачи hotfix.
- Для быстрого восстановления UX достаточно fallback-триггера поверх существующей схемы.

## Затрагиваемые файлы

### `src/components/chat/ChatWindow.tsx`

- Добавить константу `TOP_LOAD_THRESHOLD_PX` (например 40).
- Добавить ref cooldown (`loadMoreCooldownUntilRef`).
- Добавить helper `tryLoadMoreByTopPosition`:
  - проверяет `scrollTop <= threshold`,
  - проверяет `hasMore && !isLoadingOlder`,
  - проверяет cooldown,
  - вызывает `handleLoadMore()`.
- Подключить helper:
  - на `scroll` scroller-элемента,
  - дополнительно после `rangeChanged` как страховку.

### `src/components/chat/ThreadWindow.tsx`

- Применить те же изменения 1:1 для тред-ленты.

## Что не меняем

- Firestore query/индексы.
- Логику `setDisplayLimit(prev + HISTORY_PAGE_SIZE)`.
- Компенсацию prepend-позиции (`pendingPrependAdjustRef`) — она остаётся как защита от прыжков.

## Поток выполнения

1. Пользователь скроллит к верху.
2. Если `startReached` не срабатывает, fallback `scrollTop` срабатывает.
3. `handleLoadMore` увеличивает `displayLimit`.
4. Snapshot приносит расширенное окно сообщений.
5. После рендера применяется существующая компенсация scroll.

## Риски и митигации

- Риск двойной догрузки: guard `isLoadingOlder` + cooldown.
- Риск цикла на верхней границе: запуск не чаще cooldown-окна.
- Риск регресса в тредах: синхронная правка `ThreadWindow`.

## Критерии приемки

1. При достижении верха история стабильно догружается.
2. Loader (`Загрузка истории...` / `Загрузка ответов...`) появляется только во время фактической догрузки.
3. После prepend не происходит скачка позиции списка.
4. При `hasMore=false` новые запросы не стартуют.
5. Поведение одинаково работает в `ChatWindow` и `ThreadWindow`.
