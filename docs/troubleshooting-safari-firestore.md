# Safari: Firestore «Fetch API cannot load … due to access control checks»

Сообщение в консоли WebKit/Safari означает, что **браузер заблокировал** кросс-origin `fetch` к `https://firestore.googleapis.com` (каналы `Listen/channel` и `Write/channel` — нормальный транспорт клиентского Firestore SDK). Это **не** то же самое, что `permission-denied` в правилах Firestore: последнее приходит из SDK приложения, а «access control checks» — на уровне загрузки запроса.

Ниже — чеклист из плана диагностики. Выполняйте по порядку.

---

## Если помогает «Разработка → Очистить кэши» (⌥⌘E)

Если после **Разработка → Очистить кэши для всех профилей** (или **Очистить кэши**) страница **сразу** начинает работать, это почти наверняка **не «поломка» Firestore на сервере**, а **несогласованный кэш в Safari**:

- в кэше мог остаться **старый HTML или JS** от прошлого деплоя, а новые чанки `/_next/static/...` уже другие (или наоборот);
- в результате подгружается **смесь версий** клиента: сетевые вызовы (в т.ч. к `firestore.googleapis.com`) ведут себя нестабильно, в консоли появляются **access control checks** или вечный спиннер;
- полная очистка кэша заставляет браузер заново взять **актуальный документ и скрипты** — всё снова согласовано.

**Что делать:** после деплоев при отладке в Safari используйте очистку кэша или **жёсткое обновление**; при повторяющейся проблеме у конечных пользователей — **Удалить данные веб‑сайта** для вашего домена (Настройки → Конфиденциальность → Управлять данными веб‑сайтов). Service Worker приложения ([`public/sw.js`](../public/sw.js)) **не кэширует** страницу через `fetch`, но общий HTTP‑кэш Safari и данные сайта всё равно могут устареть.

**Исправление в деплое (LighChat):** для документов и прочих ответов задаётся **`Cache-Control: no-store`**, а для хэшированных чанков Next — **`public, max-age=31536000, immutable`**, чтобы HTML всегда подтягивался свежим, а статика оставались быстрой. См. [`next.config.js`](../next.config.js) и [`firebase.json`](../firebase.json). После изменения нужен **новый деплой** хостинга.

## 1. Контекст страницы и сравнение с Chrome (`context`)

- [ ] **URL**: приложение открыто только по **HTTPS** или **`http://localhost`** (для локальной разработки)? На незащищённом `http://` с произвольного хоста возможны ограничения для части веб-API и нестабильное поведение.
- [ ] **Режим Safari**: обычное окно, **приватное** или PWA «На экран «Домой»»? Приватный режим и отдельные ограничения PWA на iOS иногда усиливают блокировки.
- [ ] **Сравнение**: тот же URL и тот же аккаунт откройте в **Chrome** или **Firefox** на той же машине. Если там всё работает, а в Safari нет — сосредоточьтесь на пунктах 2–5.

---

## 2. Настройки приватности Safari (`safari-privacy`)

- [ ] Временно ослабить **«Предотвращать межсайтовое отслеживание»** (или добавить исключение для вашего домена) и перепроверить.
- [ ] Отключить **расширения** (контент-блокировщики), затем перезагрузить страницу.
- [ ] Проверить **iCloud Private Relay** (macOS/iOS): при сбоях с Google API иногда помогает временно отключить или проверить в другой сети.
- [ ] Меню **Разработка → Экспериментальные функции**: если включали нестандартные пункты — сбросить для теста.

---

## 3. Firebase Console (`firebase-console`)

- [ ] **Authentication → Settings → Authorized domains**: в списке должны быть прод-домен приложения, `localhost` (и при необходимости стейджинг). Убедитесь, что пользователь заходит **с того же хоста**, который реально открыт в адресной строке (**`www`** vs корень домена — это разные origins).
- [ ] Правила Firestore при этом сообщении обычно **не первичный** подозреваемый; если в другом браузере на том же URL всё ок — Console для Auth/domains всё равно стоит сверить.

---

## 4. Консоль Network и Service Workers (`network-sw`)

- [ ] **Network** (Safari Web Inspector): запросы к `firestore.googleapis.com` — статус (failed / cancelled / blocked) или 200? Ищите рядом в консоли упоминания **CORS**, **SSL**, **blocked**.
- [ ] **Service Workers**: в проекте PWA регистрируется [`public/sw.js`](../public/sw.js) — в нём **нет** глобального обработчика `fetch`, только FCM. Убедитесь, что нет **второго** SW с другим scope или устаревшей регистрации. При необходимости одноразово: снять регистрацию SW и сделать жёсткое обновление страницы.

---

## 5. Сеть и VPN (`corp-network`)

- [ ] Повторить сценарий **без VPN** и без корпоративного **прокси** (другая сеть или мобильный интернет). Иногда MITM или политика сети ломают длинные streaming-запросы.

## Справка по репозиторию LighChat

| Тема | Где смотреть |
|------|----------------|
| COOP для OAuth-попапов | [`next.config.js`](../next.config.js) — заголовок `Cross-Origin-Opener-Policy: same-origin-allow-popups`. К обычным Firestore `fetch` это обычно не относится. |
| Service Worker | [`public/sw.js`](../public/sw.js) — только фоновые уведомления FCM. |
| Конфиг Firebase (client) | [`src/firebase/config.ts`](../src/firebase/config.ts) — `projectId` и хосты должны совпадать с проектом в консоли. |

## Связанная проблема: Firebase Storage (обои чата)

Если в консоли **`firebasestorage.googleapis.com`**, `Cannot load image` и фигурирует **Origin … web.app** — это настройка **CORS на бакете** Storage (в коде для сэмплирования цвета фона используется `img.crossOrigin = 'anonymous'`). См. [firebase-storage-cors.md](firebase-storage-cors.md).

## Когда имеет смысл смотреть код и зависимости

- Ошибка **во всех браузерах** на одном и том же URL — проверять сеть, версию `firebase`, прокси, нестандартные обёртки над `fetch`.
- Ошибка **только в Safari** на конкретной версии ОС — искать регрессии WebKit + обновления Safari/macOS/iOS.
