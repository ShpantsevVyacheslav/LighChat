Проверь документацию проекта на синхронизацию с текущими изменениями кода.

## Алгоритм

1. Определи, какие файлы были изменены в текущей задаче (`git status`, `git diff`, `git log <base>..HEAD`).

2. Для каждого изменённого файла определи, затрагивает ли он архитектурно значимые аспекты:
   - Новые модули, директории, зоны ответственности
   - Новые/изменённые доменные сущности, типы, DTO
   - Изменения коллекций/индексов/правил Firestore
   - Новые/изменённые runtime-потоки (auth, chat, calls, meetings, notifications, games)
   - Новые/изменённые внешние интеграции, env-переменные, деплой
   - Изменения инженерных правил работы агентов

3. Применяй матрицу обновления из AGENTS.md:
   - Пути / модули / ответственность → `AGENTS.md` + `docs/arcitecture/01-codebase-map.md`
   - Типы / сущности / DTO → `docs/arcitecture/02-domain-entities.md`
   - Коллекции / связи / индексы / правила → `docs/arcitecture/03-firestore-model.md` (+ `05-integrations.md`)
   - Пользовательские / системные потоки → `docs/arcitecture/04-runtime-flows.md`
   - Внешние сервисы / env / деплой → `docs/arcitecture/05-integrations.md`
   - Правила работы агентов → `AGENTS.md` + `docs/arcitecture/06-agent-change-policy.md`

4. Обнови соответствующие документы, соблюдая стиль соседних записей:
   - `04-runtime-flows.md`: `   **Жирный заголовок (platform):** плотный абзац с [` + "`" + `ссылками` + "`" + `](../../path).`
   - `01-codebase-map.md`: `  - ` + "`" + `path` + "`" + ` - краткое описание ответственности.`
   - `05-integrations.md`: секционные буллеты с техническими деталями (TTL, ENV, ключи кеша, лимиты).

5. Верификация:
   - Проверь, что все пути в `[ссылках](...)` реально существуют (`ls`/`find`).
   - Если затронуты Firestore-правила, убедись, что `firestore.rules` и `src/firestore.rules` остаются согласованными.
   - Новые записи на русском языке, в одном стиле с соседними.

## Если изменения НЕ затрагивают архитектуру

Сообщи: «Архитектурно значимых изменений не обнаружено, документация актуальна.»
